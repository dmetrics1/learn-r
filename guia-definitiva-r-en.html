What is R and what is it for?

R is a programming language and a free, open-source environment designed specifically for statistical analysis, data science, and visualization. It is not "just another stats program": it is a full programming language whose raw material is data.

ConceptHistoryEcosystemR vs Python

A little history (just the essentials)

The story of R begins before R. In the 1970s, John Chambers and his team at Bell Laboratories created S, a language designed so statisticians could explore data interactively instead of writing rigid Fortran programs. S was revolutionary, but its commercial version (S-PLUS) was expensive and closed-source.

In 1993, Ross Ihaka and Robert Gentleman, professors at the University of Auckland (New Zealand), released a free implementation inspired by S. They named it R: after their own initials and as a nod to its predecessor. In 1995 they released it under the GPL license (free software), and since 1997 it has been maintained by the R Core Team, a group of some twenty elite statisticians and developers, alongside an enormous worldwide community. That same year saw the birth of CRAN, the official package repository.

The most recent stable release is R 4.6.0 ("Because it was There", April 2026). R ships one major version per year plus interim patches, with remarkable stability: well-written code from ten years ago usually runs today without changes.

Why the history matters: R was designed by statisticians, for analyzing data. That heritage explains its design decisions: the interactivity, vectors as the basic unit, and statistics built in from the start. Python was born as a general-purpose language and "learned" data later; R was born for it.

What makes R unique

  • Native vectorization. In R you operate on entire columns of data in one stroke, without writing loops. It is the same mental model as Excel ("apply the formula to the whole column"), but programmable and auditable.
  • Statistics out of the box. Regressions (lm()), hypothesis tests (t.test()), distributions, and frequency tables come included without installing anything. In other languages that requires external libraries.
  • Publication-quality graphics. With ggplot2 you produce figures ready for a consulting report, a paper, or a newspaper.
  • CRAN: ~22,000 reviewed packages. Every CRAN package goes through daily automated checks on several platforms. If a statistical technique has been published, there is almost certainly a package that implements it.
  • Reproducible reports. With Quarto and R Markdown, the report and the analysis live in the same document: when the data changes, the entire PDF/Word/HTML regenerates.

See vectorization in action with a classic economics case: adjusting a series of salaries for a negotiated raise.

R
salaries <- c(2.4, 3.1, 1.8)   # salaries in millions of pesos
salaries * 1.09                # a 9% raise for EVERYONE at once
[1] 2.616 3.379 1.962

No loop, no cell to drag: R applied the operation to every element. Whether you have 3 salaries or 3 million, the code is identical.

The ecosystem: the map in four names

  • CRAN — the official package repository. Installing from it is as simple as install.packages("name").
  • Posit (formerly RStudio, PBC) — the company that develops RStudio and Quarto and funds much of the tidyverse. RStudio is free and open source; Posit makes its living from enterprise products.
  • tidyverse — a collection of packages (dplyr, ggplot2, tidyr, readr…) with a shared grammar for manipulating and visualizing data. You will use it heavily throughout this guide.
  • Bioconductor — a repository specialized in bioinformatics and genomics, with more than 2,000 packages. If you come from economics you may never touch it, but it explains why R dominates the biosciences.

Don't try to memorize the ecosystem: for now it is enough to distinguish R (the language), CRAN (where packages come from), and RStudio/Posit (where you work). The rest will fall into place on its own as you move forward.

Your first code, line by line

R
# Everything after # is a comment: R ignores it completely
print("Hello, R")                # prints text to the console
2 + 2                            # R works as a calculator
sales <- c(120, 150, 90)         # creates a vector with 3 values and SAVES it as "sales"
mean(sales)                      # computes the average of the vector
[1] "Hello, R" [1] 4 [1] 120

Let's break down what just happened:

  • Line 1: the # marks a comment. R does not execute it; it is a note for humans.
  • Line 2: print() is a function: it takes something inside parentheses (its argument) and does something with it. The [1] in the output means that line starts at element 1 of the result.
  • Line 3: a bare expression is evaluated and displayed. R is interactive: you ask, it answers.
  • Line 4: two key ideas here. c() combines values into a vector, and <- (the assignment operator) saves the result under the name sales. Notice that this line prints nothing: assignment is silent. The object is now in memory, ready to use.
  • Line 5: mean() takes the whole vector and returns its average: (120 + 150 + 90) / 3 = 120.

Key idea: in R, almost everything is a vector and almost everything is done with functions. Internalize those two ideas and the rest of the language falls into place naturally: a data frame is a set of vectors, a model is fitted with a function, a chart is built with functions applied to vectors.

Best practice: comment your code from day one, explaining the why (not the obvious). When you pick up an analysis six months later —or a colleague inherits it— those comments will be worth gold.

Installing R and RStudio

You need two pieces: R (the language's engine) and RStudio (the cockpit you pilot it from). Both are free, they install in that order, and in about 10 minutes you are ready to go.

The engine-and-cockpit analogy is not just decoration: R is the interpreter that actually runs your code —you could use it from a bare terminal— while RStudio is a development environment (IDE) that gives you an editor, a console, a plot viewer, and built-in help. If tomorrow you preferred a different cockpit (VS Code, for instance), the engine would still be the same R. For learning, RStudio is the standard choice and the one this guide uses.

Step 1 — Install R (from CRAN)

Download R only from cran.r-project.org, the official repository. Choose your operating system and follow the instructions in its tab:

Path: Download R for Windows → base → Download R 4.6.x for Windows. Run the .exe and accept the default options, with one important exception:

  • Install to a path with no accents, ñ characters, or odd spaces, for example C:\R\R-4.6.0. Paths like C:\Users\José María\... or OneDrive-synced folders cause cryptic errors when installing packages.
  • Since R 4.2 only the 64-bit version exists; if a tutorial mentions 32-bit, it is outdated material.

What about RTools? It is a set of compilers for Windows. You don't need it today. It is only required when a package must be compiled from source (R will ask "Do you want to install from sources...?" — answer No and it will use the precompiled binary). Install it later only if a package demands it.

First identify your chip: Apple menu → About This Mac. Then download the correct .pkg from CRAN:

  • Apple Silicon (M1, M2, M3, M4…): the arm64 package.
  • Intel: the x86_64 package. If you install the arm64 build on an Intel Mac, it won't start.

CRAN also recommends XQuartz (xquartz.org): some graphics packages use it for certain windows and devices. Install it once and forget about it.

The r-base in your distribution's repos usually runs several versions behind. To get R 4.6 on Ubuntu/Debian, add the official CRAN repository:

bash
# Ubuntu: add the CRAN repository and its key, then install
sudo apt update
sudo apt install --no-install-recommends software-properties-common dirmngr
wget -qO- https://cloud.r-project.org/bin/linux/ubuntu/marutter_pubkey.asc | \
  sudo tee -a /etc/apt/trusted.gpg.d/cran_ubuntu_key.asc
sudo add-apt-repository "deb https://cloud.r-project.org/bin/linux/ubuntu $(lsb_release -cs)-cran40/"
sudo apt install r-base r-base-dev

For Fedora, openSUSE, or other distributions, follow the specific instructions on CRAN's Linux page. Also install r-base-dev (or its equivalent): on Linux many packages are compiled from source.

Step 2 — Install RStudio Desktop (Posit)

Download RStudio Desktop (free) from posit.co and install it with the default options. On startup, RStudio automatically finds and detects your R installation; there is nothing to configure.

Correct order: install R first and RStudio afterwards. RStudio is only the interface; it does not include the language. If you do it the other way around, RStudio will open with an "R not found" error.

To be clear about what each installer provides:

InstallerWhat it puts on your machine

RStudio (Posit)The editor with syntax highlighting and autocomplete, the integrated panes, the project manager, and viewers for plots, data, and help. Zero statistics: all of that lives in R.

Step 3 — Verify the installation

Open RStudio and type into the console (the pane with the > symbol):

If you see version 4.6.x, everything is in order. sessionInfo() will also be your best ally when asking for help: pasting its output into a forum tells anyone exactly what environment you are running.

Plan B: Posit Cloud (R in the browser)

If you work on a locked-down corporate machine, a borrowed computer, or you simply want to start right now, use Posit Cloud: it is the full RStudio running in the browser, with a free plan that covers the entire basic level. You create an account, open a "New Project", and get exactly the same interface you will see in the next section.

Tip for teams and classrooms: Posit Cloud is also great for standardizing: everyone works with the same R version and the same packages, without fighting heterogeneous local installations.

The RStudio interface

RStudio organizes your work into four panes. Understanding what each one does —and how code flows between them— will save you hours and give you professional analyst habits from day one.

The four panes

PaneWhat it is forDetail that matters

The real workflow

The panes are not four isolated compartments: they form a circuit your code flows through. The cycle you will repeat hundreds of times a day is:

  • 1. You write one or more lines in the Source (the script).
  • 2. You run the line under the cursor with Ctrl+Enter (no need to select it).
  • 3. You watch the result in the Console.
  • 4. If you assigned something with <-, the object appears in Environment; if you plotted, the figure appears in Plots.
  • 5. You fix or continue in the script, and start over.
R
# Type this in a script (Source) and run it line by line with Ctrl+Enter
monthly_cpi <- c(0.62, 0.45, 0.71, 0.38)  # CPI % change, Jan-Apr
mean(monthly_cpi)                          # monthly average
sum(monthly_cpi)                           # approximate cumulative inflation
[1] 0.54 [1] 2.16

After the first line, look at the Environment pane: you will see monthly_cpi with its four values. That is the complete circuit: Source → Console → Environment.

Why you should never work only in the console

The console is an ephemeral conversation: whatever you type there is lost when the session closes. The script is the document: the complete, reproducible recipe of your analysis. The professional rule is simple: the console is for trying things; the script is the truth. Everything you want to keep —loading data, cleaning, computing, plotting— must live in the script, so that running it top to bottom reproduces the entire analysis. When a client asks for "the same report but with this quarter's data", you will swap the input file, run the script, and be done. Had you worked in the console, you would have to rebuild everything from memory.

Warning sign: if you have typed more than two or three "keeper" commands directly into the console, stop and move them into a script. A result you cannot reproduce is a result you do not have.

RStudio Projects (.Rproj): your unit of work

A Project (File → New Project → New Directory) is a folder that RStudio treats as a self-contained unit, marked by an .Rproj file. The central benefit is the working directory: when you open the project, R automatically positions itself in that folder, so you can read data with relative paths like "data/sales.csv" instead of absolute paths like "C:/Users/luis/Documents/consulting/client/data/sales.csv".

Why does this matter so much? Portability. The absolute path only exists on your machine; the relative one works on your colleague's, on the client's server, and on Posit Cloud. You zip the project folder, send it, and the analysis runs as-is. On top of that, each project keeps its own history and session, so you can switch between client A's project and client B's without cross-contamination.

Best practice from day one: one analysis = one project. And avoid setwd() with absolute paths: it is the classic way to write code that only works on your own computer. The project handles the working directory for you.

Recommended settings (and why)

In Tools → Global Options → General, make two changes:

  • Uncheck "Restore .RData into workspace at startup".
  • Set "Save workspace to .RData on exit" to Never.

The reason: with that option on, R saves all your objects when you quit and restores them when you open. It sounds convenient, but it is a trap: your analysis comes to depend on hidden state accumulated over weeks —objects you no longer know how you created— and the day you switch machines, nothing will work. Turning it off forces your script to contain everything needed to rebuild the analysis from scratch. It is the number-one habit of reproducibility.

The habit's companion: restart the session often (Ctrl+Shift+F10) and rerun your script from the top. If something breaks after the restart, you caught a hidden dependency early —not the night before the deadline.

Customization: make it your space

In Tools → Global Options → Appearance you can pick a dark theme (many people use Tomorrow Night or similar for long sessions), increase the font size, and change the typeface (a monospaced font like Fira Code or JetBrains Mono improves code readability). In Pane Layout you can rearrange the panes if you prefer, say, the console on the right. None of this changes how your code runs; it does change how many hours your eyes can take.

Data types

Every value in R belongs to a type, and the type decides which operations are valid: you can add two numbers, but not a number and a piece of text. Mastering the type system is your vaccine against half the errors you will see in real-world data analysis.

The mental model: the type sets the rules of the game

Think of the type as each value's "ID card": it tells R what can be done with it. A salary stored as a number (2450000) can be averaged; the same salary stored as text ("2,450,000", typical of a badly exported Excel file) cannot. When a calculation fails or returns something absurd, the first question is always: what type are my data?

Why do "numeric" and "double" look like the same thing? Historically R calls the class numeric and the internal type double (64-bit double precision). In practice they are two names for the same thing; class(3.14) says "numeric" and typeof(3.14) says "double".

integer vs. double: why isn't 0.1 + 0.2 equal to 0.3?

Doubles are stored in binary with finite precision, and many decimals (like 0.1) have no exact binary representation, just as 1/3 has no exact decimal representation. The result: minuscule errors that break exact comparisons.

R
0.1 + 0.2 == 0.3                  # surprise!
print(0.1 + 0.2, digits = 20)     # what R actually stores
print(0.3, digits = 20)

sqrt(2)^2 == 2                    # same phenomenon
sqrt(2)^2 - 2                     # the error is tiny, but it exists
[1] FALSE [1] 0.30000000000000004441 [1] 0.29999999999999998890 [1] FALSE [1] 4.440892e-16

Integers, by contrast, are exact but have a ceiling: the maximum is .Machine$integer.max (2,147,483,647, about 2.1 billion). For large monetary amounts or GDP figures, use double: it gives up theoretical exactness in the decimals, but handles enormous magnitudes without overflowing.

R
.Machine$integer.max
2147483647L + 1L    # integer overflow
2147483647 + 1      # as a double, no problem
[1] 2147483647 [1] NA Warning message: NAs produced by integer overflow [1] 2147483648

Never compare decimals with ==: floating-point error is not an R bug, it is a limitation of every computer (Python, Excel and Stata suffer from it too). In the operators section you will see the correct alternatives: all.equal() and dplyr::near().

typeof() vs. class(): two different questions

typeof() asks "how is this stored in memory?" (the base type). class() asks "how should this object behave?" (the label functions use to decide what to do with it). For simple values they usually agree, but for richer objects they diverge.

R
typeof(3.14);  class(3.14)     # double / numeric
typeof(5L);    class(5L)       # integer / integer
typeof(TRUE);  class(TRUE)     # logical / logical

# Where they diverge: a date is STORED as a number
# (days since 1970-01-01) but BEHAVES like a date
today <- Sys.Date()
typeof(today)
class(today)
[1] "double" [1] "numeric" [1] "integer" [1] "integer" [1] "logical" [1] "logical" [1] "double" [1] "Date"

Implicit coercion: the hierarchy R applies without warning

A vector in R can only hold one type. If you mix types, R does not complain: it converts ("promotes") everything to the most flexible type following a fixed hierarchy, always in the direction that loses no information:

Hierarchylogical→ integer→ double→ character
Promotion exampleTRUE1L1"1"
WhyEvery TRUE is a valid 1Every integer is a valid doubleEvery double can be written as textText accepts everything (and there is no way back)
R
c(TRUE, 1L)          # logical + integer -> integer
c(1L, 2.5)           # integer + double  -> double
c(1, "two")          # double + character -> character (goodbye, arithmetic!)

# The useful coercion: TRUE counts as 1, FALSE as 0
above_minimum <- c(TRUE, FALSE, TRUE, TRUE, FALSE)
sum(above_minimum)   # how many earn above minimum wage
mean(above_minimum)  # proportion above it
[1] 1 1 [1] 1.0 2.5 [1] "1" "two" [1] 3 [1] 0.6

Analyst's trick: sum() on a logical vector counts the TRUEs and mean() computes the proportion. mean(salaries > 2000000) gives you the percentage of people earning more than two million directly, with no filtering or manual counting.

Explicit coercion: the as.*() functions

When you decide the conversion yourself, use the as.*() family. If the conversion is impossible, R does not make things up: it returns NA and (in the numeric case) warns you.

R
as.numeric("3.5")        # text that "looks like a number": works
as.integer(3.9)          # truncates, does NOT round!
as.character(100)        # number to text
as.logical(0)            # 0 is FALSE, any other number is TRUE
as.numeric("not avail.") # impossible -> NA with a warning
as.logical("hello")      # impossible -> NA (this one WITHOUT a warning)
[1] 3.5 [1] 3 [1] "100" [1] FALSE [1] NA Warning message: NAs introduced by coercion [1] NA

Best practice: when importing data, if a column that should be numeric arrives as character, that is a sign it carries junk (text like "N/A", thousands separators, currency symbols). Before blindly applying as.numeric(), inspect with unique() which values are contaminating it: every coercion warning is a data point silently turning into NA.

Common mistake: comparing against NA with ==. x == NA always returns NA, never TRUE or FALSE, because R reasons: "if I don't know its value, I can't know whether it's equal either". Always use is.na(x). Subtle detail: is.na(NaN) is TRUE (every NaN counts as missing), but is.nan(NA) is FALSE.

Finally, NA has a type. Plain NA is logical (typeof(NA) returns "logical"), and typed versions exist: NA_integer_, NA_real_ and NA_character_. You will need them in type-strict functions such as dplyr::if_else(), which requires both branches to return the same type: if_else(income > 0, income, NA_real_).

Variables and objects

A variable is not a box containing a value: it is a label attached to an object. In R everything you create (a number, a table, a regression model) is an object, and names are how you find it later.

Assignment: four forms, one convention

R
salary <- 2500000           # R's idiomatic operator
cpi = 5.2                   # valid, but reserved for another use
0.0925 -> interest_rate     # rightward assignment (rare)
assign("base_year", 2018)   # programmatic assignment (advanced)

salary
interest_rate
[1] 2500000 [1] 0.0925

Why <- and not =? Both assign, but the community reserves = for passing arguments to functions: in mean(x, na.rm = TRUE), that = creates no na.rm variable. Using <- for variables and = for arguments makes code readable at a glance: you know what is an assignment and what is an argument. RStudio shortcut: Alt+-.

What happens under the hood: names pointing to objects

When you write salary <- 2500000, R creates the object 2500000 in memory and sticks the label salary on it. If you then write new_salary <- salary, nothing gets duplicated yet: both labels point to the same object. Only when you modify one of them does R make a copy (copy-on-modify). Practical consequence: modifying a "copy" never alters the original.

R
salary <- 2500000
new_salary <- salary               # second label on the same value
new_salary <- new_salary * 1.12    # a 12% raise

salary        # the original stays intact
new_salary    # only the label you modified changed
[1] 2500000 [1] 2800000

Mental model: variables are labels on objects, not boxes with things inside. The same object can carry several labels, and peeling off one label (rm()) does not destroy the others. This model will save you when working with large data frames: passing a table to a function does not copy it until the function tries to modify it.

Naming rules (and what is forbidden)

  • A valid name starts with a letter (or a dot not followed by a number) and contains only letters, numbers, . and _. That is why cpi_2024 works and 2024_cpi does not.
  • R is case-sensitive: Age, age and AGE are three different variables (and a classic source of "object not found").
  • Reserved words cannot be used as names: if, else, for, while, function, TRUE, FALSE, NULL, NA, Inf, NaN… Trying TRUE <- 1 throws an immediate error.
  • With backticks you can create "illegal" names like `interest rate`, but you will have to type the backticks every time. Avoid it.
R
TRUE <- 1     # reserved words are locked down
T <- 0        # ...but their shortcuts T and F are NOT!
Error in TRUE <- 1 : invalid (do_set) left-hand side to assignment

Careful with T and F: they are just variables that default to TRUE and FALSE, and anyone (including you) can overwrite them: T <- 0 is legal, and from that point on any code using T is corrupted. Always write TRUE and FALSE in full.

The tidyverse style guide — the de facto standard — recommends snake_case: lowercase words separated by _. A good name saves the reader a comment:

Operators

Operators are the verbs of the language: they calculate, compare and combine conditions. Here they all are, including the two almost nobody explains well (%% and %in%) and the three traps that sooner or later bite every analyst.

R
7 + 3     # 10        addition
7 - 3     # 4         subtraction
7 * 3     # 21        multiplication
7 / 3     # 2.333333  division (always returns a double)
7 %% 3    # 1         modulo: the REMAINDER of dividing 7 by 3
7 %/% 3   # 2         integer division: the quotient without decimals
2 ^ 3     # 8         exponentiation

The modulo %% is the most underrated of the group. Three real uses:

R
# 1) Even and odd: the remainder mod 2 is 0 or 1
c(2020, 2021, 2022, 2023) %% 2 == 0

# 2) Cycles: convert a running month count into a calendar month (1-12)
cumulative_month <- c(13, 18, 24, 25)
(cumulative_month - 1) %% 12 + 1

# 3) Decompose numbers: last digit and "the rest"
code <- 4589
code %% 10     # last digit (useful for check digits)
code %/% 10    # the number without its last digit
[1] TRUE FALSE TRUE FALSE [1] 1 6 12 1 [1] 9 [1] 458

And %/% shines for grouping: (month - 1) %/% 3 + 1 turns any month into its quarter.

R
5 == 5    # TRUE   equality (double ==, not to be confused with =)
5 != 3    # TRUE   not equal
5 > 3     # TRUE   greater than
5 <= 2    # FALSE  less than or equal
"a" < "b" # TRUE   strings compare alphabetically

# Comparisons are vectorized: they compare element by element
salaries <- c(1300000, 2450000, 3100000, 1800000)
salaries > 2000000
[1] TRUE [1] TRUE [1] TRUE [1] FALSE [1] TRUE [1] FALSE TRUE TRUE FALSE

The trap in this group: == with decimals. Because of the floating-point precision you saw in data types, two calculations "equal on paper" can differ at the 16th digit:

R
1.1 - 0.7 == 0.4                    # FALSE, painful as it is
all.equal(1.1 - 0.7, 0.4)           # TRUE: compares with tolerance
isTRUE(all.equal(1.1 - 0.7, 0.4))   # the safe form for use inside if
dplyr::near(1.1 - 0.7, 0.4)         # the tidyverse's vectorized equivalent
[1] FALSE [1] TRUE [1] TRUE [1] TRUE
R
TRUE & FALSE    # FALSE  vectorized AND: both must hold
TRUE | FALSE    # TRUE   vectorized OR: one is enough
!TRUE           # FALSE  negation
xor(TRUE, TRUE) # FALSE  exclusive OR: one or the other, but not both

# In data filters, & and | work element by element
age <- c(17, 25, 34, 68)
employed <- c(FALSE, TRUE, TRUE, FALSE)
age >= 18 & employed      # employed and of working age
[1] FALSE [1] TRUE [1] FALSE [1] FALSE [1] FALSE TRUE TRUE FALSE

The double versions && and || are scalar: they demand a single value on each side and are used in if conditions. They also short-circuit: if the left side already decides the result, the right side is never evaluated.

R
(5 > 3) && (2 < 4)      # TRUE: both scalar conditions

# Short-circuit as a guard: if x is NULL, x > 0 is never evaluated
x <- NULL
!is.null(x) && x > 0    # FALSE without error; with & it would have failed

# Since R 4.3, && with vectors of length > 1 is an ERROR (it used
# to silently take the first element: worse)
c(TRUE, FALSE) && TRUE
[1] TRUE [1] FALSE Error in c(TRUE, FALSE) && TRUE : 'length = 2' in coercion to 'logical(1)'

%in% asks whether each element on the left belongs to the set on the right. It is the elegant replacement for endless chains of == glued together with |:

R
dept <- c("Magdalena", "Antioquia", "Atlántico", "Bolívar", "Cesar")

# Clumsy and fragile:
dept == "Magdalena" | dept == "Atlántico" | dept == "Bolívar"

# Clear and scalable:
dept %in% c("Magdalena", "Atlántico", "Bolívar")

# Bonus: %in% handles NA sensibly, == does not
income_na <- c(100, NA, 300)
income_na == 100          # the NA propagates
income_na %in% 100        # %in% always answers TRUE/FALSE
[1] TRUE FALSE TRUE TRUE FALSE [1] TRUE FALSE TRUE TRUE FALSE [1] TRUE NA FALSE [1] TRUE FALSE FALSE

Rule of thumb: if you are comparing against two or more values, or if your vector may contain NA, use %in%. In data filters (the Caribbean region, the months of a quarter, the ISIC codes of interest) it is the operator you will use most.

One vs. two: & and | operate element by element on vectors (for filtering data); && and || return a single value and are used in if conditionals. Since R 4.3, applying && or || to vectors longer than 1 throws an error, which is a blessing: it used to silently take the first element and hide bugs.

Classic mistake: using = instead of ==. x = 5 assigns; x == 5 compares. Inside an if, the first usually raises a syntax error, but inside a function call it can slip through as an argument and produce baffling behavior.

Precedence: why -2^2 is -4

When an expression mixes operators, R applies them in a fixed order, not left to right. From highest to lowest priority:

Best practice: do not memorize the full table: whenever an expression combines three or more operators, use parentheses even if they are redundant. ((year %% 4) == 0) & ((year %% 100) != 0) reads effortlessly and removes all ambiguity, for R and for the colleague who inherits your code.

Comparing money: if you work with monetary figures that carry multiplications by rates (interest, deflators, exchange rates), assume == will fail you at some point. To check that a reconciliation balances, prefer abs(a - b) < 0.01 (one-cent tolerance) or dplyr::near(a, b, tol = 0.01).

Vectors

The vector is THE central structure of R: an ordered sequence of values of the same type. It is not just one structure among many: in R, everything else (matrices, factors, data frames) is built on top of vectors. Understanding vectors is, quite literally, understanding R.

Here is the key mental model: in R there are no scalars. When you type 5, R treats it as a numeric vector of length 1. That is why printing it shows [1] 5: that [1] tells you "this is the element in position 1". Every R function is designed with vectors in mind, and that explains why the language feels so natural for data analysis: a column of sales, a series of prices, or the answers to a survey are vectors.

Creating vectors: c(), sequences, and repetitions

The c() function (for combine) is the most used in all of R. But for regular sequences there are more expressive tools:

R
sales <- c(150, 172, 168, 195)    # c() = combine values
1:6                                # integer sequence in steps of 1
seq(0, 1, by = 0.25)               # you control the step
seq(0, 100, length.out = 5)        # you control how many elements
seq_len(4)                         # 1 2 3 4 (safe for loops)
seq_along(sales)                   # 1 2 3 4 (one index per element)
[1] 1 2 3 4 5 6 [1] 0.00 0.25 0.50 0.75 1.00 [1] 0 25 50 75 100 [1] 1 2 3 4 [1] 1 2 3 4

To repeat values, rep() has two arguments people mix up: times repeats the whole block; each repeats each element before moving on to the next. Compare them side by side:

R
rep(c("T1", "T2"), times = 3)   # the whole block, 3 times
rep(c("T1", "T2"), each = 3)    # each element, 3 times in a row
rep(c("A", "B"), times = c(2, 4))  # different repetitions per element
[1] "T1" "T2" "T1" "T2" "T1" "T2" [1] "T1" "T1" "T1" "T2" "T2" "T2" [1] "A" "A" "B" "B" "B" "B"

seq_along() will save you later on: when you write loops, use for (i in seq_along(x)) instead of for (i in 1:length(x)). If x is empty, 1:length(x) produces 1 0 (the loop runs twice with invalid indices!), whereas seq_along(x) produces an empty vector and the loop never runs. It is the difference between a robust script and a silent bug.

The 5 ways to index (it starts at 1!)

Indexing means asking the vector for a subset using square brackets [ ]. Mastering all five ways is essential because they are exactly the same ones you will later use with matrices and data frames:

R
sales <- c(jan = 150, feb = 172, mar = 168, apr = 195)

sales[1]              # 1. positive position: "give me element 1"
sales[c(2, 4)]        # ...or several at once
sales[-1]             # 2. negative position: "everything EXCEPT 1"
sales[sales > 165]    # 3. logical condition: "the ones that qualify"
sales["mar"]          # 4. by name
which(sales > 165)    # 5. which(): at which POSITIONS does it hold?
jan 150 feb apr 172 195 feb mar apr 172 168 195 feb mar apr 172 168 195 mar 168 feb mar apr 2 3 4

Logical indexing deserves a moment of your attention: sales > 165 produces the vector FALSE TRUE TRUE TRUE, and when you use it inside [ ] R keeps only the positions with TRUE. This pattern — build a condition and filter with it — is the heart of every data filter you will ever write in R, including filter() from the tidyverse.

Very common mistake: in R, indices start at 1, not at 0 (unlike Python or C). v[0] does not throw an error: it returns an empty vector of length 0, which is worse because the bug goes unnoticed. And asking for a position that does not exist, like v[10] on a vector of 4, returns NA without complaint.

Named vectors: names(sales) returns "jan" "feb" "mar" "apr", and you can also assign names afterwards with names(v) <- c(...). A named vector works like a mini dictionary: it is the simplest way to pair a label with a value without needing a data frame yet.

Vectorization: R's superpower

In most languages, applying VAT to 4 prices would mean writing a loop. In R, operations act on the whole vector at once. Compare the same calculation done both ways:

R
prices <- c(100, 250, 80, 120)
with_vat <- numeric(length(prices))       # empty vector of the right size
for (i in seq_along(prices)) {
  with_vat[i] <- prices[i] * 1.19
}
with_vat
[1] 119.0 297.5 95.2 142.8
R
prices <- c(100, 250, 80, 120)
with_vat <- prices * 1.19    # one single line, no indices, no loop
with_vat
[1] 119.0 297.5 95.2 142.8

Same result, but the vectorized version is shorter, more readable, and much faster: the "loop" happens internally in compiled C code, not in R. The why matters: when you think "I need to do X to every element", your first reflex in R should not be a loop, but asking yourself whether the operation is already vectorized. It almost always is: +, *, sqrt(), log(), round(), comparisons, paste()... they all work element by element.

R
costs <- c(80, 190, 60, 95)
margin <- prices - costs        # operation between two vectors, pairwise
margin
margin / prices * 100           # percentage margin of each product
[1] 20 60 20 25 [1] 20.00000 24.00000 25.00000 20.83333

Recycling: the rule and the danger

What happens if you operate on two vectors of different lengths? R recycles the shorter one: it repeats it until it reaches the length of the longer one. That is what makes prices * 1.19 work (the 1.19, a vector of length 1, gets recycled 4 times). The rule: if the longer length is an exact multiple of the shorter one, R recycles silently; if it is not, R recycles anyway but issues a warning.

R
c(1, 2, 3, 4) + c(10, 20)   # 4 is a multiple of 2: recycles without a word
c(1, 2, 3) + c(10, 20)      # 3 is NOT a multiple of 2: recycles AND warns
[1] 11 22 13 24 [1] 11 22 13 Warning message: In c(1, 2, 3) + c(10, 20) : longer object length is not a multiple of shorter object length

The real danger of recycling is not the warning, but the case where there is NO warning. If you accidentally add a 12-month vector to a 4-quarter vector, the lengths line up (12 is a multiple of 4), R says nothing, and you get plausible but wrong numbers. When results look suspicious, check lengths with length().

Matrices

A matrix is not a new structure: it is a vector with a dim attribute added (rows × columns). It is still a single sequence of values of the same type, only R displays and indexes it in two dimensions. It is the foundation of linear algebra in R.

You can verify that "matrix = vector + dimensions" yourself: take a vector of 6 numbers and assign it dimensions.

R
v <- 1:6
dim(v) <- c(2, 3)   # the vector becomes a 2x3 matrix!
v
is.matrix(v)        # TRUE
[,1] [,2] [,3] [1,] 1 3 5 [2,] 2 4 6

Creating: by columns (default) or by rows

The usual way is matrix(). The detail that causes the most surprises: R fills the matrix by columns (it goes down the first column, then the second...). If your data arrives "read by rows", use byrow = TRUE:

R
matrix(1:6, nrow = 2)                 # fills by columns (default)
matrix(1:6, nrow = 2, byrow = TRUE)   # fills by rows
[,1] [,2] [,3] [1,] 1 3 5 [2,] 2 4 6 [,1] [,2] [,3] [1,] 1 2 3 [2,] 4 5 6

Why by columns? Because internally the matrix is still a vector, and R (like Fortran, its numerical ancestor) stores it column after column. Knowing this also explains why m[3] — with a single index — returns the third element of the underlying vector, not the third row.

Row and column names

A matrix with names documents itself. Let's work with an economic case: units sold of two products over three months.

R
sales <- matrix(c(120,  90, 150,
                  135, 110, 160),
                nrow = 2, byrow = TRUE)

Indexing with [row, column] and the drop trap

The syntax is m[row, column]; leaving a slot empty means "all of them". The same five indexing methods you learned with vectors apply here (positions, negatives, logicals, names, which()).

R
sales[1, ]        # complete first row... but it returns a VECTOR!
sales[, "mar"]    # the March column, also as a vector
sales[1, , drop = FALSE]   # row 1 KEEPING the matrix structure
jan feb mar 120 90 150 coffee cocoa 150 160 jan feb mar coffee 120 90 150

The lost-dimension trap: when you extract a single row or column, R "simplifies" the result to a vector and the matrix loses its dimensions. In interactive analysis it is harmless, but inside a function or loop it breaks code that expects a matrix (for example, something that later calls nrow() or %*%). The safeguard is drop = FALSE. File this detail away: it comes back with data frames.

Element-wise vs matrix multiplication

Here lives the most expensive mistake in this section. * between matrices multiplies cell by cell; the matrix product of linear algebra (rows by columns) is the %*% operator:

R
A <- matrix(c(1, 2, 3, 4), nrow = 2)   # [1 3; 2 4] (remember: by columns)
A * A       # element-wise: each cell squared
A %*% A     # MATRIX product: rows x columns
t(A)        # transpose
[,1] [,2] [1,] 1 9 [2,] 4 16 [,1] [,2] [1,] 7 15 [2,] 10 22 [,1] [,2] [1,] 1 2 [2,] 3 4

And solve(A) computes the inverse. Verify it against the definition: a matrix times its inverse gives the identity.

R
A <- matrix(c(2, 1, 1, 3), nrow = 2)
solve(A)            # inverse
A %*% solve(A)      # must give the identity
[,1] [,2] [1,] 0.6 -0.2 [2,] -0.2 0.4 [,1] [,2] [1,] 1 0 [2,] 0 1

Careful: confusing * with %*% does not always throw an error: if the dimensions are compatible, you get a matrix of wrong numbers with no warning at all. In econometrics (think of solve(t(X) %*% X) %*% t(X) %*% y, the OLS formula) that silence destroys the result. When in doubt, verify one cell by hand.

Row and column summaries

The rowSums(), colSums(), rowMeans(), colMeans() family summarizes the matrix along one dimension, and they are much faster than any loop:

R
rowSums(sales)    # quarter total per product
colMeans(sales)   # average sales for each month
coffee cocoa 360 405 jan feb mar 127.5 100.0 155.0

Factors

Factors represent categorical variables: qualitative data with a finite, known set of categories (levels), such as region, education level, or socioeconomic stratum. They are R's answer to a question every analyst faces: how do I tell the software that "primary, secondary, university" are not mere strings, but categories with meaning?

Why do factors exist?

Three concrete reasons:

1. Modeling. When you pass a factor to a regression (lm(salary ~ region)), R automatically creates the dummy variables and picks the reference category. With plain text you would have to build them by hand.
2. Order. A character vector does not know that "low" comes before "high"; an ordered factor does, and that carries over to tables, plots, and comparisons.
3. Domain control. A factor knows its valid categories: if you try to assign a value outside the levels, R turns it into NA and warns you — a free data-quality check. (Memory efficiency, the historical argument, is secondary today: modern R also compresses repeated strings.)

How they are stored internally: integers + labels

This is the mental model that prevents every scare: a factor is a vector of integers (the codes 1, 2, 3...) plus a levels attribute holding the label for each code. Prove it:

R
sizes <- factor(c("M", "S", "L", "M", "S"))
sizes
typeof(sizes)     # internally it is integer!
unclass(sizes)    # the bare codes + their levels
[1] M S L M S Levels: L M S [1] "integer" [1] 2 3 1 2 3 attr(,"levels") [1] "L" "M" "S"

Notice two things: the levels were sorted alphabetically by default (L, M, S — not the logical order S, M, L), and each value in the vector is actually the number of its level. That is why "M" is stored as 2. Almost every problem with factors comes from forgetting this dual nature.

levels and labels in factor(): control order and labels

levels declares which values exist in the input data and in what order; labels defines how you want to display them. They are distinct and combinable — very handy when the data comes coded as numbers, which is typical in surveys:

R
# The survey codes stratum as 1, 2, 3
stratum <- factor(c(1, 3, 2, 1, 2, 2),
                  levels = c(1, 2, 3),
                  labels = c("low", "medium", "high"))
stratum

# Fix the alphabetical order of sizes to the logical order
sizes <- factor(sizes, levels = c("S", "M", "L"))
levels(sizes)
[1] low high medium low medium medium Levels: low medium high [1] "S" "M" "L"

Reordering the reference for regression: in a model, the factor's first category is the baseline for comparison. relevel(region, ref = "Caribe") moves "Caribe" to first place without touching anything else — so the lm() coefficients are interpreted "relative to the Caribbean". To reorder everything, call factor(x, levels = ...) again.

Ordered factors: when categories have a hierarchy

With ordered = TRUE, R understands that the levels have a real order and enables comparisons:

R
educ <- factor(c("secondary", "primary", "university",
                 "secondary", "secondary", "university"),
               levels = c("primary", "secondary", "university"),
               ordered = TRUE)
educ[1] < educ[3]         # secondary < university?
educ[educ >= "secondary"] # filter by minimum level
[1] TRUE [1] secondary university secondary secondary university Levels: primary < secondary < university

Look at how it prints: primary < secondary < university. With an unordered factor, those comparisons return NA with a warning, because it makes no sense to ask whether "Caribe" is less than "Andina".

Frequency tables and proportions

R
table(educ)                        # count per level
round(prop.table(table(educ)), 3)  # proportions (they sum to 1)
educ primary secondary university 1 3 2 educ primary secondary university 0.167 0.500 0.333

A valuable note: table() on a factor shows all the levels, even those with zero frequency. With a character vector, a category with no cases simply vanishes from the table — with factors, you see the zero, which is often the interesting fact.

Ghost levels and droplevels(): when you filter data, unused levels persist (with frequency 0 in tables and empty gaps in plots). droplevels(f) removes levels with no observations. It is routine after any subsetting: sub <- droplevels(educ[educ != "primary"]).

The classic trap: as.numeric() on a factor

Now that you know a factor is integer codes, this trap explains itself. If a factor contains numbers-as-text and you convert it directly to numeric, R gives you back the internal codes, not the values:

R
f <- factor(c("10", "50", "20", "50"))
as.numeric(f)                  # WRONG! internal codes: positions in levels
as.numeric(as.character(f))    # RIGHT: first to text, then to number
as.numeric(levels(f))[f]       # idiomatic equivalent (and more efficient)
[1] 1 3 2 3 [1] 10 50 20 50 [1] 10 50 20 50

Why it is so dangerous: as.numeric(f) throws no error and no warning, and returns perfectly plausible numbers. If your income column arrived as a factor (it happened all the time when importing CSVs with older R) and you convert it the wrong way, every downstream analysis is silently corrupted. Golden rule: factor → character → numeric, never the direct jump.

An honest preview: forcats in the tidyverse

When you get to the tidyverse, the forcats package (for for categorical variables) will give you more convenient versions of everything above. The two you will use most:

R
library(forcats)
fct_infreq(educ)                   # reorder levels by frequency (ideal for bar charts)
fct_reorder(region, mean_income)   # reorder one categorical by another variable

The logic does not change: they are still factors with codes and levels. forcats simply makes ergonomic what you just learned to do by hand — which is why it pays to understand the base mechanics first.

Best practice: define the levels explicitly when creating any important factor, even if the alphabetical order happens to match the one you want. It documents your intent, makes the ordering in tables and plots reproducible, and catches misspelled values (a typo like "Secondry" will become NA and you will see it immediately).

Lists

The list is R's universal container: it can hold elements of different types and lengths — vectors, matrices, factors, functions, and even other lists. If the vector is R's building brick, the list is the box that fits any combination of bricks.

Why devote an entire section to something so simple? Because almost everything complex in R is a list in disguise: the result of a regression (lm), the output of a statistical test (t.test), a JSON read from an API, and even the data frame you will see in the next section (which is a list of equal-length columns). Knowing how to navigate lists means knowing how to extract results from any analysis.

Creating a list

R
customer <- list(
  name      = "Luis",
  age       = 28,
  active    = TRUE,
  purchases = c(45000, 120000, 89000),   # a vector inside the list
  contact   = list(email = "luis@mail.com", city = "Santa Marta")  # a nested list!
)
length(customer)   # 5 elements, regardless of each one's size
names(customer)
[1] 5 [1] "name" "age" "active" "purchases" "contact"

[ vs [[ vs $: the train analogy

Picture the list as a freight train: each element is a car with something inside.

list[1] — single bracket — gives you the car hitched to a shorter train: the result is still a list (of one element). Use it to sub-select the train.
list[[1]] — double bracket — opens the car and hands you the cargo: the element itself, with its original type.
list$name — the named shortcut — does the same as [[, but writing the name without quotes.

R
customer["purchases"]     # a 1-car train: still a LIST
customer[["purchases"]]   # the cargo: the numeric vector
customer$purchases        # identical to [[, more convenient
class(customer["purchases"]); class(customer$purchases)
$purchases [1] 45000 120000 89000 [1] 45000 120000 89000 [1] 45000 120000 89000 [1] "list" [1] "numeric"

The #1 confusion with lists: trying to operate on list[1]. mean(customer["purchases"]) fails ("argument is not numeric") because you are handing it a sealed car — a list — and not the numbers. If the goal is to use the content, always [[ ]] or $. If the goal is to keep a sub-list, [ ].

Nested access and modification

For lists within lists, you chain accesses from left to right, like opening boxes inside boxes:

R
customer$contact$city           # contact box -> city element
customer[["purchases"]][2]      # second value of the purchases vector

customer$age <- 29              # modify an element
customer$segment <- "premium"   # add a new one (just assign it)
customer$active <- NULL         # delete: assign NULL
names(customer)
[1] "Santa Marta" [1] 120000 [1] "name" "age" "purchases" "contact" "segment"

One-line preview: lapply(list, f) applies the function f to each element and returns another list: lapply(list(a = 1:3, b = 4:6), mean) returns $a = 2 and $b = 5. It is the bridge between lists and iteration, and you will study it in depth in the functions level.

Why they matter: a regression model IS a list

Let's fit the classic regression on the cars dataset (stopping distance as a function of speed) and open up the result as what it really is — a list with 12 components:

R
model <- lm(dist ~ speed, data = cars)
names(model)             # the model's "train cars"
model$coefficients       # the (named) vector of coefficients
model$coefficients["speed"]   # just the slope
head(model$residuals, 3)      # first residuals
[1] "coefficients" "residuals" "effects" "rank" [5] "fitted.values" "assign" "qr" "df.residual" [9] "xlevels" "call" "terms" "model" (Intercept) speed -17.579095 3.932409 speed 3.932409 1 2 3 3.849460 11.849460 -5.947766

Everything the model knows is right there, reachable with the same tools you just learned: $, [[, indexing by name. When a tutorial tells you "extract the model's residuals", you now know it simply means opening one car of the list. (In practice there are also extractor functions like coef(model) and residuals(model), which are the recommended route — but they work precisely because there is a list underneath.)

The same pattern: API/JSON-style data

When you read data from an API (with jsonlite, for example), the JSON turns into... a nested list. The structure is identical to what you already know how to navigate:

R
response <- list(
  status = 200,
  data = list(
    user = list(id = 501, name = "Luis", city = "Santa Marta"),
    purchases = c(45000, 120000, 89000)
  )
)
response$data$user$city           # go down three levels
sum(response$data$purchases)      # total purchased
response[["data"]][["user"]][["id"]]   # equivalent with [[ ]]
[1] "Santa Marta" [1] 254000 [1] 501

str(): your flashlight for unfamiliar lists

Faced with someone else's list (a model, a JSON, the object a package returns), don't guess: str() (structure) shows you the full map — names, types, sizes, and nesting. With max.level you control how many levels deep to look:

R
str(response, max.level = 2)
List of 2 $ status: num 200 $ data :List of 2 ..$ user :List of 3 ..$ purchases: num [1:3] 45000 120000 89000

Best practice: the professional workflow for any unknown object is always the same: str(x, max.level = 1) to see the map, names(x) to know the cars, and then $ or [[ ]] to extract exactly what you need. It works the same for an lm, a t.test, or an API response.

Connection to what's next: a data frame is, technically, a list whose elements are vectors of the same length (the columns). That is why df$column uses the same $ syntax you just learned: it is no coincidence — it is the same operation on the same structure.

Data Frames

The data frame is the star structure of data analysis: a table where each column is a variable and each row is an observation. It looks like a spreadsheet, but with one key difference: understanding what it is on the inside will save you half of the mistakes of your first year with R.

The mental model: a list of vectors

A data frame is a list whose elements are vectors of the same length. Each column is a vector; each vector is a column. This sentence sounds trivial, but it explains almost everything about how a data frame behaves:

  • Each column has exactly ONE type (numeric, character, logical…), because a vector is homogeneous. That is why, if the text "N/A" sneaks into a salary column, the entire column turns into text.
  • Rows, on the other hand, can mix types: a row is not a vector — it is a "cross-section" of several vectors.
  • Everything you learned about vectors (vectorized operations, comparisons, NA) applies directly to columns.
R
df <- data.frame(
  name = c("Ana", "Beto", "Caro"),
  age  = c(25, 31, 28),
  city = c("Cali", "Bogotá", "Medellín")
)
df
typeof(df)   # on the inside...
class(df)    # ...on the outside
length(df)   # 3! the "length" of a data frame is its COLUMNS
name age city 1 Ana 25 Cali 2 Beto 31 Bogotá 3 Caro 28 Medellín [1] "list" [1] "data.frame" [1] 3

Mental model: think of the data frame as a filing cabinet with vertical folders (the columns). Each folder holds a single type of document, and all of them contain the same number of sheets (the rows). typeof(df) returns "list" because that is literally what it is.

Scouting the terrain: the first 5 commands

Whenever you receive a data frame (yours or someone else's), do the standard reconnaissance before computing anything. It is the equivalent of walking the lot before building on it. Let's use a small employee register from a consulting firm:

Filtering rows: simple and combined conditions

The base-R recipe is df[condition, columns]. The condition is a logical vector built from the columns, and you can combine conditions with & (and) and | (or):

R
# Employees with 5+ years of tenure, name and area only
employees[employees$tenure >= 5, c("name", "area")]

# Combined condition: in IT AND with a high salary
employees[employees$area == "IT" & employees$salary > 4600, ]

# Logical OR: in Finance OR with more than 6 years
employees[employees$area == "Finance" | employees$tenure > 6, ]
name area 2 Beto IT 4 David Finance 5 Elena IT name area salary tenure 5 Elena IT 4800 5 name area salary tenure 2 Beto IT 4500 7 4 David Finance 5100 10

The readable base-R alternative is subset(): it takes the data frame, the condition (without repeating employees$) and, optionally, the columns:

R
subset(employees, salary > 4000, select = c(name, salary))
name salary 2 Beto 4500 4 David 5100 5 Elena 4800

The NA trap: ghost rows

Here comes a classic that will happen to you sooner or later. Fabio's salary is NA. What does NA > 3000 return? Neither TRUE nor FALSE: it returns NA ("I don't know"). And when you index a data frame with an NA, R does not drop the row — it hands you a ghost row filled with NA:

R
employees[employees$area == "Sales" & employees$salary > 3000, ]
name area salary tenure 1 Ana Sales 3200 4 NA <NA> <NA> NA NA

Two clean fixes: wrap the condition in which() (which ignores NAs and returns only the positions that are truly TRUE) or exclude them explicitly with !is.na(). subset() also drops them on its own:

R
employees[which(employees$area == "Sales" & employees$salary > 3000), ]
employees[!is.na(employees$salary) & employees$salary > 3000, ]  # explicit
name area salary tenure 1 Ana Sales 3200 4

Ghost rows: if rows full of NA show up when you filter, it is not an R bug: there are NAs in the column you filtered by. Use which() or !is.na(). In a survey database with thousands of missing values, this detail changes your counts and your averages.

Creating, modifying and deleting columns

Since every column is a vector, creating a column is simply assigning a vector (or a vectorized operation on other columns). Deleting one is assigning NULL:

R
employees$annual_salary <- employees$salary * 12         # new, calculated
employees$senior        <- employees$tenure >= 5         # new, logical
employees$salary[3]     <- 3100                          # modify ONE value
names(employees)[1]     <- "employee"                    # rename a column
employees$annual_salary <- NULL                          # delete a column
head(employees, 3)
employee area salary tenure senior 1 Ana Sales 3200 4 FALSE 2 Beto IT 4500 7 TRUE 3 Caro Sales 3100 2 FALSE

Sorting with order()

Mind the distinction: sort() sorts the values of a vector, but to sort a data frame you need order(), which returns the positions in the right order. Those positions are then used as the row index:

R
# From highest to lowest salary (NAs end up last)
employees[order(employees$salary, decreasing = TRUE), ]

# Two criteria: area alphabetically and, within each area, salary descending
employees[order(employees$area, -employees$salary), ]
employee area salary tenure senior 4 David Finance 5100 10 TRUE 5 Elena IT 4800 5 TRUE 2 Beto IT 4500 7 TRUE 1 Ana Sales 3200 4 FALSE 3 Caro Sales 3100 2 FALSE 6 Fabio Sales NA 1 FALSE

Practice without hunting for data: R ships with built-in datasets: mtcars (cars), iris (flowers), airquality (air quality, with real NAs). Type data() to see the full catalog. We will use them throughout this guide because they are available in every R installation.

With four building blocks (data frame, vector, logical condition, tapply) you have already answered a real business question. At Level 2, dplyr will do this same job with a more fluent grammar — but the underlying logic is exactly this.

Importing and exporting data

90% of real projects don't start with clean data: they start with a file somebody else put together — a CSV exported from an accounting system, an Excel file with three title rows, an official statistics download with broken accents. Importing well is your first professional R skill.

The good news: R reads practically any format. The bad news: every format has its traps, and almost all of them are predictable. This section gives you the tools plus the catalog of classic disasters so that none of them catches you off guard.

For CSV you have two families: read.csv() from base R (always available) and read_csv() from the readr package (tidyverse). For serious projects, readr is the better choice: it is noticeably faster on large files, returns a tibble, doesn't mangle column names, and it shows you the type it guessed for each column — that report is your first audit, free of charge.

R
# Base R: always available
data <- read.csv("sales.csv")

# readr (tidyverse): faster, more informative
library(readr)
data <- read_csv("sales.csv")
Rows: 1250 Columns: 4 ── Column specification ───────────────────────── Delimiter: "," chr (2): region, product dbl (1): amount date (1): date

Always read that report: if amount shows up as chr instead of dbl, the file contains decimal commas, currency symbols or a hidden "N/A". For non-standard separators there is read_delim():

R
data <- read_csv2("sales.csv")     # ; separator and decimal comma (LatAm/Europe)
data <- read_delim("log.txt", delim = "|")   # any separator
data <- read_csv("sales.csv", skip = 2)      # real header on row 3

Real-life Excel files come with cover pages, merged titles and multiple sheets. readxl has an argument for each of those ailments. Explore the sheets first, then read with precision:

R
library(readxl)

[1] "Cover" "Sales" "Notes"

Useful notes: range overrides skip (if you give a range, the skip is ignored); and in col_types the value "skip" drops a column right at read time — perfect for junk columns.

R also talks to the entire statistical and web ecosystem:

Why does RDS matter? It is the only format that preserves everything: exact types, factors with their levels, dates, attributes, even a complete regression model. A CSV stores nothing but plain text — when you re-read it, R has to guess the types all over again. That is why the professional workflow is: raw data in CSV/Excel, intermediate results in RDS.

Bonus for survey work (household surveys, censuses): haven keeps SPSS/Stata labels as attributes; with as_factor() you turn them into R factors.

Rule of thumb: if the recipient is a person using Excel, use write_xlsx() or write_excel_csv(). If the recipient is your own script tomorrow, use saveRDS(). If it is another system, standard CSV.

The European/Latin American problem: ; and the decimal comma

In Colombia and much of Latin America — as in most of Europe — Excel uses the comma as the decimal separator (1.234,56). Since the comma is already taken, when exporting to "CSV" Excel separates fields with semicolons. The result: a file called .csv that is not comma-separated at all. If you read it with read_csv(), you get one giant column or numbers turned into text.

R
# The one-line fix for the semicolon-separated CSV
data <- read_csv2("sales_2025.csv")

# Fine control with locale(): odd separator + decimal comma + dot for thousands
data <- read_delim("sales_2025.txt", delim = "|",
                   locale = locale(decimal_mark = ",", grouping_mark = "."))

The locale() object is readr's regional-settings box: decimal mark, thousands separator, encoding, time zone and month names for dates in Spanish (locale("es")).

Encoding: when accented characters break

If after importing you see BogotÃ¡, MedellÃ­n or Ã± where there should be an ñ, you have an encoding problem: the file is saved in Latin-1 (typical of Excel and older Windows systems) but R is reading it as UTF-8, the modern standard. Each accented character occupies different bytes in each encoding, and misreading them produces those hieroglyphics.

R

The return trip breaks too: if you export a CSV in UTF-8 and your client opens it in Excel, they will see broken accents. write_excel_csv() adds a marker (BOM) that tells Excel the file is UTF-8 — problem solved in both directions.

Paths: use projects, not C:/Users/...

An absolute path like "C:/Users/luis/Documents/project/data.csv" works on exactly one computer on the planet: yours, today. The moment you share the script with a colleague, move it to another machine or reorganize your folders, everything breaks. here() builds the path from the project root no matter where the script is run from — it works the same on your laptop, on the client's machine or on a server. One extra line that buys you total reproducibility.

Mistake #1 when importing: absolute paths. If your script contains "C:/Users/...", it is not reproducible: it breaks on any other machine (and on yours, six months from now). RStudio project + here::here("data", "sales.csv"), always.

na.rm = TRUE in depth: why NA contaminates

NA means "unknown value". And arithmetic with the unknown is relentlessly honest: if you don't know how much March sold, you cannot know the quarter's total. That is why 10 + NA is NA, and a single missing value contaminates any complete sum or average. The argument na.rm = TRUE ("NA remove") tells the function: drop the missing values and compute with what is there.

R
v <- c(10, NA, 30)
mean(v)                # NA  ← a single NA contaminates everything
mean(v, na.rm = TRUE)  # 20  ← mean of the values that are present
sum(is.na(v))          # 1   ← how many are missing? ALWAYS ask this
[1] NA [1] 20 [1] 1

Almost every summary function accepts it: sum, mean, median, sd, var, min, max, range, quantile. A fine point for when you get to correlations: cor() does not use na.rm but use = "complete.obs".

Extremely common error: getting NA when computing an average or a total. It is not a bug: there are missing data. But before adding na.rm = TRUE on autopilot, count the NAs with sum(is.na(x)). An average computed on 40% of the data is a different figure from one computed on 98% — and your report should say so.

Locating and checking: which, any, all

These functions turn logical conditions into concrete answers: where? (which), any? (any), all? (all). Plus a central trick of analysis in R: since TRUE equals 1 and FALSE equals 0, sum() over a condition counts and mean() computes the proportion.

R
x <- c(4, 8, 15, 16, 23, 42)
which(x > 15)       # at which positions?
which.max(x)        # position of the maximum (the first one, in case of ties)
which.min(x)        # position of the minimum
any(x > 40)         # is there at least one?
all(x > 3)          # do they all pass?
sum(x > 15)         # count how many pass
mean(x > 15)        # proportion that passes
[1] 4 5 6 [1] 6 [1] 1 [1] TRUE [1] TRUE [1] 3 [1] 0.5

The analyst's golden pattern: sum(condition) to count and mean(condition) for proportions. "What percentage of customers bought more than $1 million?" is one line: mean(purchases > 1e6, na.rm = TRUE).

Text: paste, paste0, sprintf and format

paste() joins pieces with a space; paste0() joins them with no separator (ideal for building names); sprintf() inserts values into a template with controlled formatting — the reporting tool; and format() adds thousands separators and pins down decimals for presenting figures:

R
paste("Region", 1:3)                       # vectorized: recycles "Region"
paste0("ind_", c("gdp", "cpi"))            # no space
paste(c("Cali", "Bogotá"), collapse = " - ")  # collapses a vector into ONE string
[1] "Region 1" "Region 2" "Region 3" [1] "ind_gdp" "ind_cpi" [1] "Cali - Bogotá"

In sprintf(), the key codes are %d (integer), %.2f (decimal with 2 digits), %s (text) and %% for a literal percent sign:

R
sprintf("Annual inflation: %.2f%%", 5.847)
sprintf("Processed %d records from %s", 1250, "the household survey")
format(45230.5, big.mark = ",", nsmall = 2)          # US/UK style
format(1500000, big.mark = ".", decimal.mark = ",")  # European/Latin style
[1] "Annual inflation: 5.85%" [1] "Processed 1250 records from the household survey" [1] "45,230.50" [1] "1.500.000"

Mnemonic rule: paste to glue, sprintf for templates with numeric formatting, format to dress up individual figures. In an automated report the three work together: you calculate at full precision and format only at the end, never before.

round() and the banker's rounding surprise

Brace yourself for a classic surprise: R does not round .5 "always up" the way you were taught in school. It follows the IEC/IEEE 754 standard: an exact .5 rounds to the nearest even number ("banker's rounding"), because always rounding up would bias the sums of millions of transactions upward:

R
round(0.5)   # 0! (the nearest even number)
round(1.5)   # 2
round(2.5)   # 2, not 3!
round(3.5)   # 4
round(2.675, 2)   # 2.67: also, 2.675 has no exact binary form (IEEE 754)
[1] 0 [1] 2 [1] 2 [1] 4 [1] 2.67

Two complementary tools: round(x, digits) fixes decimal places (and accepts negative digits to round to thousands), while signif(x, digits) fixes significant figures — ideal for reporting economic magnitudes without false precision:

R
round(18726.4, -3)    # to thousands
signif(18726.4, 2)    # 2 significant figures
signif(0.004321, 2)   # works the same at small scales
ceiling(4.1); floor(4.9); trunc(-4.7)   # trunc cuts toward zero
[1] 19000 [1] 19000 [1] 0.0043 [1] 5 [1] 4 [1] -4

If a total is "off by a cent" compared with Excel or the accountant's report, suspect banker's rounding and binary arithmetic (IEEE 754). It is not an R error: it is the standard. To reconcile accounting figures, round once, at the very end, and document the criterion.

Putting it all together: cleaning and summarizing monthly sales

Help and documentation

Nobody memorizes all of R: professionals consult documentation every single day. Knowing where to look, how to read a help page, and how to decipher an error is a skill just as important as programming itself, and you train it like any other.

Think about it: the difference between a beginner who gets stuck and one who keeps moving is rarely how much they know by heart. It's that the second one turns every error into an effective search. In this section you'll learn that entire workflow: built-in documentation, reading errors, searching online, and asking for help (from humans or an AI) in a way that gets you good answers.

Anatomy of a help page

Every documented function in R has a help page with the same structure. Open it with ? followed by the name:

R
?mean          # opens the help page for mean() in the Help pane
help(mean)     # exactly the same, in function form
example(mean)  # runs the help page's examples in your console
mean> x <- c(0:10, 50) mean> xm <- mean(x) mean> c(xm, mean(x, trim = 0.10)) [1] 8.75 5.50

Help pages feel intimidating at first because they're designed as reference, not as a tutorial. The trick is knowing which section to check depending on what you need:

SectionWhat it containsWhen to check it
DescriptionWhat the function does, in 1–3 sentences.First. Confirm it's the right function before reading further.
UsageThe signature: arguments in order and their default values.To see what you can omit. If an argument has = value, that's its default.
ArgumentsEach argument explained: expected type and meaning.When an argument doesn't do what you expected. This is where the fine detail lives.
ValueWhat the function returns and its type.Key when you're going to store the result: is it a vector, a list, a data frame?
ExamplesRunnable code that demonstrates the function.Always. It's the most valuable section on the entire page.

Examples is pure gold: it's real code, written by the function's author, that runs without changing a thing. Copy it, run it line by line, and tweak values to see what happens. Ten minutes playing with the examples teach you more than an hour reading the theory, because you see concrete inputs and outputs.

The built-in search tools

Each tool answers a different question. Using them in the right order saves a lot of time:

R
?median                    # "I know the exact function name"
??regression               # "I don't know the name; search ALL installed docs for the word"
help.search("regression")  # formal equivalent of ??
args(read.csv)             # "I just need a quick reminder of the arguments"
vignette("dplyr")          # long-form tutorial written by the package authors
browseVignettes()          # lists ALL vignettes from your installed packages
> args(read.csv) function (file, header = TRUE, sep = ",", quote = "\"", dec = ".", fill = TRUE, comment.char = "", ...) NULL

Notice the key difference: ? demands the exact name (?regresion finds nothing), while ?? does a fuzzy search by topic across every package you have installed. And vignettes are the perfect middle ground: more narrative than a help page, shorter than a book. When you install a new package, skimming its main vignette is the fastest way to understand its philosophy.

How to read an error message

An R error is not a scolding: it's a diagnosis with a fixed structure. Learning to take it apart turns panic into information. The general anatomy is:

Error in FUNCTION_THAT_FAILED(...) : text explaining the problem

R
sum("10")        # ERROR: stops execution
salees           # ERROR: misspelled name (typo)
log(-1)          # WARNING: it runs, but the result is suspicious
mean(c(2, "4"))  # WARNING: returns NA and tells you why
Error in sum("10") : invalid 'type' (character) of argument Error: object 'salees' not found [1] NaN Warning message: In log(-1) : NaNs produced [1] NA Warning message: In mean.default(c(2, "4")) : argument is not numeric or logical: returning NA

Break down the first one the way an analyst would: Error in sum("10") tells you which function failed (sum) and with what input; invalid 'type' (character) of argument is the key text: you passed text to a function that expects numbers. The second error doesn't even mention a function; object '...' not found almost always means a misspelled name or an object you haven't created yet. And when the error happens inside a chain of functions and you can't tell which one failed, running traceback() right afterwards shows you the full call stack.

Error, warning, and message are not the same thing. An error stops everything: there is no result. A warning lets execution continue, but the result may be garbage (like the NA above): never ignore one without understanding it. A message is just information (like the notices when loading a package). Treating warnings as "errors that got lucky" will save you from silently wrong analyses.

Searching Google and Stack Overflow like a professional

When the documentation isn't enough, you search the error online. But there's technique to it:

  • Search in English and add in R at the end: invalid 'type' (character) of argument in R. 95% of the answers are in English; searching in other languages drastically shrinks your results.
  • Strip out anything specific to your data. Nobody else has a column called sales_march_2026. Search the generic part of the message, not your variable names or file paths.
  • Copy the literal message in quotes for an exact search, and without quotes if nothing comes up.
  • If your R shows errors in another language, you can run Sys.setenv(LANGUAGE = "en") at the start of the session so they come out in English and are easier to search.

The reprex: your calling card

If after searching you decide to ask, the quality of your question determines the quality (and speed) of the answer. The currency of R communities is the reprex: a minimal reproducible example. It must meet three conditions: include data anyone can generate (or a built-in dataset like iris), the minimal code that produces the problem, and the exact error. That way, whoever reads it can reproduce your problem in 10 seconds, which multiplies your chances of getting an answer.

R
# BAD: "my average doesn't work, help" + a blurry screenshot

# GOOD: a reprex — runs on any machine and shows the problem
ages <- c(23, 31, NA, 45)
mean(ages)            # expected 33, I get NA. Why?
#> [1] NA

# The reprex package formats all of this (code + output) ready to paste:
install.packages("reprex")
reprex::reprex(mean(c(23, 31, NA, 45)))

Golden rule when asking for help: the effort you put into the question comes back to you in the answer. A good reprex also has a magical side effect: by shrinking your problem down to its minimum, half the time you find the bug yourself before you even post.

  • #RStats on social media: the R community is remarkably welcoming.

And always verify by running the code: an AI can invent functions or arguments that don't exist, and the only final judge is your R console. If you can't explain the code you turned in, it's not yours yet.

Step 0 · What an EDA is and why it's always the first step

An EDA is the phase where you get to know your data before modeling it or drawing conclusions: what it contains, what shape it's in, how it's distributed, and what relationships it hides. Skipping it is the number one cause of failed analyses: models trained on duplicates, averages distorted by missing values, conclusions about columns that weren't even numeric. Every serious EDA starts from a guiding question. Ours:

Do sepal and petal measurements let us tell the three iris species apart?

That question decides what we look at in each step. Without a question, an EDA is just a list of commands.

Structured programming

Up to now you typed loose commands at the console: one line, one result, another line. That works for exploring, but not for real work. Structured programming is the leap from "pressing buttons" to "building a machine": a file that decides, repeats, and reuses logic, one you can run again tomorrow without standing over it.

From the loose command to the script

A loose command lives in your head and in your working memory. You know you loaded the data first, then filtered, then averaged... but that order only exists as long as you remember it. A script makes that order explicit: it is the written recipe, not the plated dish. The practical difference is enormous.

  • Reproducibility: someone else (or you in three months) gets the same result by running the file.
  • Auditability: every number in the report has a line that produced it. Nothing comes "out of nowhere."
  • Scale: changing the year of analysis means editing one parameter, not redoing fifty steps by hand.

The mental model: think of a script as an assembly line. Raw data enters one end, each station transforms it a little, and out the other end comes a finished result. If a station fails, the line stops right there — and that is good, because it tells you exactly where to look.

The three pillars

All of structured programming, in any language, is built from just three ways of controlling flow. Everything else is a combination of these three.

PillarWhat it doesIn RMetaphor
SequenceRuns step after step, in orderOne line below anotherFollowing a recipe top to bottom
SelectionPicks a path based on a conditionif, switch, case_when()A fork in the road: if it rains, I grab the umbrella
IterationRepeats until something is metfor, while, the apply familyWhisk until the mixture thickens

Sequence is the one you already use without thinking. The other two pillars are the subject of the coming sections. For now, hold on to the idea: controlling flow is deciding what runs, when, and how many times.

Anatomy of a reproducible script

A professional script is not code thrown together at random; it has a recognizable structure any analyst can read at a glance. The order of the blocks is no whim: it mirrors how information flows, from the general to the specific, from raw to finished.

R · sales_analysis.R
# ============================================================
# Monthly sales analysis — Caribbean Region
# Author: L. Durango   ·   Last edited: 2026-07-08
# Goal: compute net sales and flag outlier months
# ============================================================

# 1. LIBRARIES ------------------------------------------------
library(tidyverse)   # everything the script uses goes up here

# 2. PARAMETERS AND PATHS ------------------------------------
outlier_threshold <- 100     # sales below this value are flagged
vat               <- 0.19
data_path         <- "data/sales_2025.csv"

# 3. LOAD -----------------------------------------------------
sales <- read_csv(data_path)

# 4. TRANSFORM -----------------------------------------------
sales <- sales |>
  mutate(net = gross * (1 - vat),
         alert = net < outlier_threshold)

# 5. ANALYSIS -------------------------------------------------
summary <- sales |>
  group_by(month) |>
  summarise(total = sum(net), outliers = sum(alert))

# 6. OUTPUT ---------------------------------------------------
write_csv(summary, "outputs/sales_summary.csv")
print(summary)

Why this order and not another? Each block answers a question the reader asks in sequence:

  • Header: "what does this do and who wrote it?" — the context that keeps you from opening the file blind.
  • Libraries up top: "what do I need installed?" — if a package is missing, you fail on line 8, not line 200.
  • Parameters together: "what can I change without breaking the logic?" — the VAT, the threshold, and the paths are knobs, not code buried somewhere. Changing the year is a one-line edit.
  • Load → Transform → Analyze → Output: the data travels in one direction. You never re-read the CSV in the middle of the analysis.

Parameters up top, magic inside: every "magic" number (a threshold, a rate, a year) should be a named variable at the start. When the boss asks "run it with a 21% VAT," you change vat <- 0.21 in one place instead of find-and-replacing across the whole file.

Idempotence: run top to bottom, always the same

A script is idempotent when running it twice produces the same state as running it once. This sounds obvious, but it breaks easily. The most common culprit is relying on state left behind by a previous run in the session.

R
# FRAGILE: if you run this line twice, the price goes up twice
price <- price * 1.19

# ROBUST: always start from the original value
price_with_vat <- base_price * 1.19

The acid test: restart the R session (in RStudio, Session → Restart R) and run the whole file from top to bottom. If it finishes without errors and gives what you expect, it is reproducible. If it depends on something that "was already loaded," it is not — and that something will vanish on the day you least expect it.

Never save the workspace: turn off "restore .RData on startup." A saved workspace gives you the illusion that everything works, when in reality your script depends on ghost objects nobody else has. The single source of truth must be the code.

source() vs. running line by line

While developing, you run line by line (Ctrl+Enter in RStudio): you test, look at the result, adjust. That is exploration. But the final verdict comes from source("analysis.R"), which runs the entire file as a block, in one pass, exactly as a server or a colleague would.

R
source("sales_analysis.R")           # runs everything silently
source("sales_analysis.R", echo = TRUE)  # shows each line as it runs

The key difference: running line by line leans on your memory and your mental order; source() forgives nothing. If a script only works "when I run it, in this order, with these variables already loaded," then it does not work. source() is the impartial referee of reproducibility.

Conditionals

A conditional is a fork in the road: the code evaluates a yes/no question and takes one route or the other. Mastering conditionals in R means distinguishing two worlds that beginners confuse: the scalar world (one decision) and the vectorized world (one decision per element).

if / else if / else: scalar decision

The if form evaluates a single condition (one TRUE or FALSE value) and runs the matching block. It is for control logic: deciding the program's flow, not classifying data.

R
grade <- 3.8

if (grade >= 4.0) {
  print("Excellent")
} else if (grade >= 3.0) {
  print("Pass")
} else {
  print("Fail")
}
[1] "Pass"

Style that reads itself: open the brace { on the same line as the if, put the else right next to the closing brace }, and indent the body. Always use braces even when the body is one line: it avoids bugs when tomorrow you add a second statement and forget the braces.

The classic mistake: "the condition has length > 1"

This is the number-one stumble when moving from scalars to real data. if expects one logical value. If you hand it a vector of several, R no longer knows which path to take: the first element's? the majority's? The correct answer is that the question is ill-posed.

R
grades <- c(4.5, 2.8, 3.2)

if (grades >= 3.0) {          # three answers for a single question?
  print("Pass")
}
Error in if (grades >= 3) { : the condition has length > 1

Since R 4.3 this is an ERROR, not a warning. Before, R silently took only the first element and carried on — a source of invisible bugs for years. Now it stops dead. If you see the condition has length > 1, your question is about a vector: you need ifelse() or case_when(), not if.

Vectorized ifelse(): one decision per element

ifelse() is the vectorized sibling. It takes a logical vector and returns a vector: for each position, it picks between the "yes" value and the "no" value. It is the right tool when you classify data columns.

R
grades <- c(4.5, 2.8, 3.2)
ifelse(grades >= 3.0, "Pass", "Fail")
[1] "Pass" "Fail" "Pass"

The golden rule: if to control the flow of the program (one decision); ifelse() to transform data (one decision per row). If the condition is a column, you almost never want if.

switch(): map a category to a value

When you have a variable that can take several discrete values and you want to assign a result to each, switch() is cleaner than a ladder of if/else if. It is ideal for rates, per-segment tariffs, or translating codes.

R
# Discount by customer type
type <- "premium"

discount <- switch(type,
  basic   = 0.00,
  premium = 0.15,
  vip     = 0.30,
  0.00)          # last unnamed argument = default value

discount
[1] 0.15

That last unnamed argument acts as a safety net: if type matches no case, switch() returns the default instead of NULL. Without it, an unknown type would return nothing and you would break the calculation further down.

Fall-through trick: if several cases share a result, leave the name empty and "fall" to the next one. For example monday = , tuesday = , wednesday = "weekday" makes all three return "weekday". Note: switch() is scalar — it takes one value, not a vector.

case_when(): multiple conditions, readable

When the conditions are not "equal to X" but ranges or combinations (income between A and B, age over C and region equal to D), the ladder of nested ifelse() becomes unreadable. dplyr::case_when() solves this: it lists condition ~ result pairs, top to bottom, and is vectorized.

R · dplyr
library(dplyr)

income <- c(800000, 2500000, 6000000, 1500000)

segment <- case_when(
  income <  1160000 ~ "below minimum wage",
  income <  4000000 ~ "middle",
  income >= 4000000 ~ "high",
  .default = "no data"        # replaces the old TRUE ~ ...
)
segment
[1] "below minimum wage" "middle" "high" "middle"

The ordering trap: case_when() evaluates top to bottom and keeps the first true condition. If you put income >= 1160000 ~ "middle" before the "high" condition, every high income will fall into "middle" because it is also greater than 1,160,000. Order from most specific/restrictive to most general.

Consistent type across all arms: every result on the right of the ~ must be the same type. Mixing "high" (text) with 0 (number) in different arms throws a type error. If an arm is NA, use the correctly typed NA: NA_character_ for text, NA_real_ for numbers.

Loops

A loop repeats instructions. It is the iteration pillar in action. But in R there is a healthy tension: often the loop you would write in another language is unnecessary, because R vectorizes. Learning loops includes learning when not to use them.

for, while, and repeat

All three repeat, but they answer different questions. for: "I know how many times." while: "I repeat as long as something holds." repeat: "I repeat and I decide when to exit."

R
# for: known number of iterations
for (i in 1:3) {
  cat("Quarter", i, "\n")
}

# while: until capital doubles the goal
capital <- 1000
years   <- 0
while (capital < 2000) {
  capital <- capital * 1.05   # 5% per year
  years   <- years + 1
}
cat("Doubles in", years, "years\n")

# repeat: infinite loop with an explicit exit
n <- 0
repeat {
  n <- n + 1
  if (n == 3) break
}
Quarter 1 Quarter 2 Quarter 3 Doubles in 15 years

break and next: break leaves the loop entirely; next jumps to the following iteration without running what remains. Think of next as "this round doesn't interest me, let's move on" and break as "I'm done, I'm out."

The antipattern: growing a vector inside the loop

This is the most common performance mistake in R. When you do result <- c(result, x) on each pass, R does not enlarge the vector: it creates a new one and copies everything from before. On iteration 1000 you are copying 999 elements just to add one. The cost grows with the square of the size.

R
# The vector is re-created and copied on EVERY pass
result <- c()
for (i in 1:100000) {
  result <- c(result, i^2)   # copies everything from before
}
# Time: grows quadratically. Painful with large data.
R
# Reserve the space ONCE and just fill it in
n <- 100000
result <- numeric(n)              # 100,000 zeros ready
for (i in seq_len(n)) {
  result[i] <- i^2               # writes into slot i
}
# Time: grows linearly. Orders of magnitude faster.

Conceptual measurement: with 100,000 iterations, the growing-vector version can take seconds or minutes; the pre-allocated one, milliseconds. The rule is simple: if you know the size of the result, reserve it beforehand with numeric(n), character(n), or vector("list", n).

seq_along and seq_len, not 1:length()

Writing for (i in 1:length(x)) hides a deadly trap: if x ends up empty, length(x) is 0 and 1:0 generates c(1, 0) — the loop runs twice with meaningless indices! Use seq_along(x), which generates exactly the valid indices and produces nothing if x is empty.

R
x <- c()
1:length(x)      # c(1, 0)  ← dangerous
seq_along(x)     # integer(0) ← empty, correct

for (i in seq_along(x)) { ... }   # runs nothing if x is empty

Loop or vectorization?

The central question in R. A loop is correct and necessary when:

  • Each iteration depends on the previous one (a series where today's balance uses yesterday's).
  • The goal is a side effect: printing, writing files, plotting at each step.
  • You iterate over heterogeneous things (different tables, different models).

By contrast, if the operation is "apply the same thing to each element independently," vectorize: it is faster and more readable.

R
# Unnecessary loop
vats <- numeric(length(prices))
for (i in seq_along(prices)) vats[i] <- prices[i] * 0.19

# Vectorized: R applies * to the whole vector at once
vats <- prices * 0.19

R's mental model: R is designed to think in whole vectors, not element by element. Addition, multiplication, comparisons — everything operates on the entire vector. When you catch yourself writing a for to transform a column, stop and ask whether there is a vectorized operation.

Writing your own functions

Writing your own functions is the step that separates someone who uses R from someone who programs in R. A function encapsulates an idea: you give it a name, test it once, and reuse it without ever thinking about its guts again. It is the basic unit for not repeating yourself and for not making the same mistake twice.

Anatomy of a function

Every function has three parts: the arguments (what goes in), the body (what it does), and the return value (what comes out). In R, a function automatically returns the value of the last expression evaluated; the explicit return() is optional but useful for early exits.

R
convert_temp <- function(celsius, scale = "F") {      # arguments
  if (scale == "F") {                                 # body
    celsius * 9/5 + 32
  } else {
    celsius + 273.15                                  # Kelvin
  }
}                                                     # last expr = return

convert_temp(25)          # 77    (Fahrenheit by default)
convert_temp(25, "K")     # 298.15
[1] 77 [1] 298.15

Implicit return vs. return(): R's idiomatic style lets the last expression be the result. Reserve return() for exiting early: for example, if (length(x) == 0) return(NA) at the start, so you do not keep computing on empty data. A return() at the end of a function is redundant.

Default arguments and ...

Default values make a function easy in the common case and flexible in the rare one. And ... (the "dots") captures extra arguments to pass them on to another inner function, without having to list them all.

R
describe <- function(x, ...) {
  c(mean = mean(x, ...),
    sd   = sd(x, ...))
}

describe(c(1, 2, NA, 4))                # NA because there is a missing value
describe(c(1, 2, NA, 4), na.rm = TRUE)  # the ... pass na.rm to mean and sd
mean sd 2.33 1.53

Lexical scope: why not to touch global variables

R uses lexical scope: a function looks for variables where it was written, not where it was called. Inside a function, assignments create local variables that vanish when it ends. This is a virtue: the function does not pollute your global environment. Modifying globals from inside breaks that promise and makes the result depend on invisible state.

R
vat <- 0.19

compute <- function(price) {
  vat <- 0.05          # local: does NOT alter the outer vat
  price * (1 + vat)
}

compute(100)   # 105
vat            # still 0.19, untouched
[1] 105 [1] 0.19

Avoid <<- to modify globals: there is a super-assignment operator that does write outward, but use it as a last resort. A function that modifies global variables is unpredictable: its result depends on what was in memory before. The right way is to receive what you need as an argument and return what you produce. In through the door, out through the door.

Validate the inputs with stopifnot()

A robust function distrusts its arguments. stopifnot() checks conditions at the start and aborts with a clear message if something is off — the "fail hard and early" principle we saw in scripts, applied to functions.

R
final_price <- function(price, discount = 0, vat = 0.19) {
  stopifnot(
    "Price must be numeric" = is.numeric(price),
    "Discount goes between 0 and 1" = discount >= 0 && discount <= 1
  )
  price * (1 - discount) * (1 + vat)
}

final_price(100000, 1.5)   # invalid discount → it stops
Error: Discount goes between 0 and 1

Fail at the edge, not at the center: validate on the function's first line. A clear error the moment bad data comes in is a thousand times better than a silent NA that contaminates the report and you discover three steps later without knowing where it came from.

Short syntax: \(x)

Since R 4.1 there is a shorthand for writing anonymous functions: \(x) is equivalent to function(x). It is ideal for one-line functions inside sapply, map, or Filter.

R
square <- \(x) x^2         # identical to function(x) x^2
square(6)                 # 36

sapply(1:4, \(x) x * 100) # 100 200 300 400

Design principles

One function, one responsibility: if your function loads data, cleans it, models it, and plots it, those are really four functions fighting over one name. Split it. Each function should do one thing and do it well; that way they are tested and reused separately.

  • Verb names: a function does something, so its name is an action — compute_vat, classify_income, clean_text. Nouns are for data.
  • Data first: the main argument (the vector, the table) goes first; the options come after, with default values. This fits the pipe |>.
  • Pure when you can: same input, same output, no hidden effects. These are the easiest to test and to trust.

Integrative example: an income classifier

Let's put it all together — default arguments, validation, vectorization, and case_when — into a function with an economic flavor that classifies monthly incomes by multiples of the minimum wage.

R · dplyr
library(dplyr)

classify_income <- function(income, min_wage = 1300000) {
  stopifnot("Income must be numeric" = is.numeric(income))
  mw <- income / min_wage                 # income in minimum wages
  case_when(
    mw <  1 ~ "below minimum wage",
    mw <  2 ~ "1 to 2x minimum wage",
    mw <  4 ~ "2 to 4x minimum wage",
    .default  = "over 4x minimum wage"
  )
}

classify_income(c(900000, 2000000, 3500000, 7000000))
[1] "below minimum wage" "1 to 2x minimum wage" "2 to 4x minimum wage" "over 4x minimum wage"

Note the design: a verb name, data first, the minimum wage as a parameter with a default (so it works for any year by changing one number), validation at the start, and vectorization to classify a whole vector at once.

Error handling

Robust code anticipates what can go wrong. When you load 300 sales files and number 187 has a comma instead of a period, you don't want the whole pipeline to crash: you want R to tell you what failed, where, and keep going. R organizes the unexpected into three conditions: errors (they halt), warnings (they alert) and messages (they inform).

The mental model: three conditions

R doesn't have a single "something happened" channel. It has a condition system: an object travels up the call stack looking for someone to catch it. Think of it as three alarm levels at a consulting firm:

ConditionRaised byHalts?Business analogy
errorstop()Yes, abortsThe contract is invalid: you can't continue.
warningwarning()No, but accumulatesA data point looks suspicious: keep going, but review it later.
messagemessage()NoInformational log: "loading quarter 3...".

A key detail that confuses everyone at first: message() and warning() do not write to normal output (stdout), but to the message channel (stderr). That's why they appear in red in the RStudio console even though they aren't errors. Don't use print() for diagnostics: use message(), so the user can silence them with suppressMessages().

Why the distinction matters: a warning lets a batch of 300 files finish and leaves you at the end with a list of the 4 that failed. A stop() kills everything at file 5. Choosing the right condition means deciding whether your process is fragile or resilient.

tryCatch in depth

By default, an error propagates upward and ends the program. tryCatch() intercepts that condition and gives you a chance to react. Its anatomy has four pieces: expr (what you attempt), and named handlers like error, warning and finally (what happens no matter what).

R
# Safe read: never halts the pipeline, returns NULL on failure
read_safe <- function(path) {
  tryCatch(
    expr = {
      data <- read.csv(path)
      message("Read: ", path, " (", nrow(data), " rows)")
      data
    },
    error = function(e) {
      message("Could not read '", path, "': ", conditionMessage(e))
      NULL
    },
    warning = function(w) {
      message("Warning while reading '", path, "': ", conditionMessage(w))
      NULL
    },
    finally = message("-- read attempt finished --")
  )
}

sales <- read_safe("sales_2026.csv")
missing <- read_safe("does_not_exist.csv")
-- read attempt finished -- Read: sales_2026.csv (1240 rows) Could not read 'does_not_exist.csv': cannot open the connection -- read attempt finished --

Three things you should burn into memory from this pattern:

  • conditionMessage(e) extracts the error text. The e object isn't a string: it's a condition with structure. Never do paste("Error:", e) directly.
  • The handler replaces the value. Whatever error = function(e) ... returns is what the whole tryCatch() returns. Here we return NULL so we can filter it out afterward.
  • finally always runs, error or not. It's where you close connections, delete temporaries or write to the log. It returns no value, it only runs side effects.

tryCatch vs try: there's also try(expr, silent = TRUE), simpler, but it returns an object of class try-error that you must inspect manually. For serious code prefer tryCatch(): it's explicit about which condition you catch.

Catching a warning cancels execution: if you define a warning= handler, the moment the first warning fires R jumps to the handler and abandons the rest of expr. If you only want to capture warnings without interrupting, use withCallingHandlers() instead of tryCatch(). It's the difference between "exit" and "log and continue".

Raising conditions: stop, warning, message and stopifnot

On the other side of the counter, you're the one writing functions others will use. Validate early and fail clearly. Compare a fragile function with one that protects the user:

R
# Computes the revenue growth rate between two periods
growth_rate <- function(base, current) {
  if (!is.numeric(base) || !is.numeric(current))
    stop("base and current must be numeric; got ",
         class(base), " and ", class(current))
  if (any(base == 0))
    warning("There are zero bases: the rate will be Inf")
  if (any(current < base))
    message("Note: some period declined.")
  (current - base) / base
}

growth_rate(c(100, 200), c(120, 180))
Note: some period declined. [1] 0.2 -0.1

stopifnot() is the shortcut for multiple validations: each expression must be TRUE, and if not, the error message tells you which one failed. Since R 4.0 you can name each check so the message is even clearer:

R
average_salary <- function(x) {
  stopifnot(
    "x must be numeric"          = is.numeric(x),
    "x cannot be empty"          = length(x) > 0,
    "x cannot have negatives"    = all(x >= 0, na.rm = TRUE)
  )
  mean(x, na.rm = TRUE)
}

average_salary(c(1500, -200, 1800))
Error in average_salary(c(1500, -200, 1800)) : x cannot have negatives

Actionable messages: a good error answers three questions: what it expected, what it got and (if you can) what to do. "x must be numeric, got character" is worth gold; "Error in sqrt" wastes half an hour. Always include the real value or class with class(x) or typeof(x).

Robust pipelines: rlang and purrr

When you process hundreds of cases, you don't want one bad one to bring down the batch. The tidyverse offers two tools for this. rlang improves error messages (with formatting, bullets and context); purrr gives you adverbs that wrap a function so it never stops the flow:

R
library(rlang)
library(purrr)

# rlang::abort — structured error with class and bullets
validate_age <- function(age) {
  if (age < 0) {
    abort(
      message = c(
        "Age cannot be negative.",
        "x" = paste0("Got: ", age),
        "i" = "Check the encoding of missing values (maybe -99?)."
      ),
      class = "error_negative_age"
    )
  }
  age
}

# purrr::safely — each call returns list(result, error), never crashes
read_all <- map(c("q1.csv", "broken.csv", "q3.csv"), safely(read.csv))

# purrr::possibly — on failure, returns a default value
rates <- map_dbl(divisors, possibly(\(d) 100 / d, otherwise = NA_real_))

With safely(), read_all is a list where each element has $result (the data or NULL) and $error (the condition or NULL). At the end you separate the good from the bad with purrr::transpose() and know exactly which files failed, without having interrupted anything.

The right adverb: use safely() when you want to inspect the errors afterward; possibly() when you just want a default value and to move on; quietly() when you need to capture warnings and output without cluttering the console.

Debugging: when it already failed

The first three debugging resources, from least to most powerful:

  • traceback(): after an error, it shows the call stack — which function called which, up to the crash point. It's the first thing you run to locate the origin.
  • browser(): insert it inside a function and execution pauses there, giving you an interactive console with the local variables live. You type n (next line), c (continue), Q (quit).
  • print-debugging: the classic message("value of x: ", x) at strategic points. Inelegant, always works.

Pro shortcut: options(error = recover) opens an interactive menu of the stack every time something fails, so you can jump to any level of execution and inspect it. And debug(f) automatically activates step-by-step mode every time f is called.

Packages and libraries

A package is code, data and documentation bundled for reuse. Base R ships the essentials; packages give it superpowers. There are more than 21,000 on CRAN, from econometrics to scraping open-data portals. Knowing how to install them, load them and pin them is what separates an analysis that works today from one that also works two years from now.

Install vs load: the number-one confusion

They are two distinct actions that happen at distinct moments, and confusing them is the most common mistake of beginners:

ActionFunctionWhen?Analogy
Installinstall.packages()Once per machineBuying the book and putting it on the shelf.
Loadlibrary()Every session you use itTaking the book off the shelf onto your desk.
R
install.packages("tidyverse")   # downloads and compiles: ONCE
library(tidyverse)               # activates the package: EVERY session

update.packages(ask = FALSE)     # update everything installed
installed.packages()[, "Package"]  # see what you have installed

Never put install.packages() in a script you share. It would reinstall everything every time someone runs it, and worse: install.packages() requires network and permissions. Install from the console; the script should only contain library().

library vs require

Both load a package, but they fail differently, and that difference matters:

  • library(dplyr): if the package isn't installed, it halts with an error. That's what you want in a script: fail loudly and early.
  • require(dplyr): it returns TRUE/FALSE and only emits a warning. It's useful inside an if to check availability, but if you use it on its own, your script will keep going with a missing package and blow up 40 lines later with an incomprehensible error.
R
# Legitimate use of require: check before continuing
if (!require("readxl")) {
  install.packages("readxl")
  library(readxl)
}

Simple rule: in scripts use library(). Reserve require() for the rare case where you need the logical value inside a condition.

The :: operator and when to use it

dplyr::filter() calls filter from the dplyr package without loading the whole package. Three uses where it's the right tool:

  • Disambiguate when two packages have the same function (we'll see this shortly).
  • Use a single function without polluting your session with the whole package: scales::comma() to format a number, without library(scales).
  • Document the origin in code others will read: lubridate::ymd() makes clear where it comes from.
R
# Use a specific function without loading the entire package
dplyr::filter(mtcars, mpg > 25)
scales::dollar(1500000)   # "$1,500,000"

Name conflicts: "last one wins"

Here's a classic trap. The stats package (base R) ships filter() for time series. dplyr ships its own filter() for filtering rows. When you load dplyr after, its filter covers the one from stats: the last loaded wins, and R warns you:

R
library(dplyr)
Attaching package: 'dplyr' The following objects are masked from 'package:stats': filter, lag

"Masked" means covered. From here on, a filter() without a prefix is dplyr's. If you wanted the one from stats, you must write stats::filter(). The conflicted package turns this silent ambiguity into an explicit error: it forces you to decide which one you want, instead of guessing.

R
library(conflicted)
library(dplyr)

filter(mtcars, mpg > 25)   # now this doesn't guess: it forces you to decide
Error: ! [conflicted] filter found in 2 packages. Either name the source explicitly: dplyr::filter stats::filter

The silent bug: without conflicted, if you load packages in a different order on another machine, "last one wins" can resolve filter to a different function and your script will do something different without erroring. That's why in serious projects it's recommended to load conflicted or prefix sensitive functions with ::.

Reproducibility with renv: the problem it solves

Imagine you deliver a salary analysis to a client. Six months later, dplyr releases version 2.0, changes the behavior of a function, and your script produces different numbers, or won't run. Worse: your colleague has ggplot2 3.4 and you have 3.5, and the chart comes out differently. That's the reproducibility crisis in miniature.

renv gives each project its own isolated library and a renv.lock file that records the exact version of every package. It's like a detailed receipt of your environment that anyone can rebuild identically:

R
install.packages("renv")
renv::init()      # creates an isolated library just for this project
renv::snapshot()  # writes the exact versions into renv.lock
renv::restore()   # on ANOTHER machine: reinstalls those identical versions
# renv.lock (excerpt) "dplyr": { "Package": "dplyr", "Version": "1.1.4", "Source": "Repository" }

The flow is: init() once when you start the project, snapshot() every time you add or update a package, and restore() for whoever receives the project. You version renv.lock in git and your analysis becomes a time machine.

Where packages live: .libPaths() shows the folders where R searches for and installs packages. Without renv, all your projects share a global library (that's why one update breaks things). With renv, each project has its own renv/library/ folder, isolated from the rest.

The tidyverse

The tidyverse is a collection of packages that share a common philosophy, grammar and data structure for data science. It's not "another library": it's a way of thinking about analysis. Learning one of its packages brings you closer to all the rest. Current version: tidyverse 2.0.

The shared philosophy

What unifies the tidyverse is three design commitments you'll see repeated in every package:

  • The data frame is at the center. Almost every function takes a data frame as its first argument and returns another. That's what makes chaining possible.

  • Consistency. Arguments, names and behaviors resemble each other across packages, so what you learn in dplyr serves you in stringr or tidyr.

Tidy data: the central concept

All the power of the tidyverse rests on a simple but profound idea about how to structure a table. A dataset is tidy when it meets three rules:

  1. Each variable is a column.
  2. Each observation is a row.
  3. Each value is a cell.

It sounds obvious, but most real-world tables don't meet it. Look at this quarterly sales table, typical of an Excel report:

R
# MESSY: the quarters are columns, not a variable
# region   Q1     Q2     Q3
# North    120    150    170
# South     90    110    130

Here "quarter" is a variable disguised as three columns, and "sales" (a single concept) is split across three cells per row. This complicates everything: how do you compute average sales per quarter? The tidy version puts each variable in its own column:

R
library(tidyverse)

sales_wide <- tribble(
  ~region,  ~Q1,  ~Q2,  ~Q3,
  "North",  120,  150,  170,
  "South",   90,  110,  130
)

# pivot_longer tidies: one row per observation (region-quarter)
sales_wide |>
  pivot_longer(cols = Q1:Q3, names_to = "quarter", values_to = "sales")
# A tibble: 6 × 3 region quarter sales <chr> <chr> <dbl> 1 North Q1 120 2 North Q2 150 3 North Q3 170 4 South Q1 90 5 South Q2 110 6 South Q3 130

Now each row is an observation (one region in one quarter), each variable has its own column and you can group, filter and plot without friction. Almost all of the tidyverse assumes this format: ggplot2 wants one column per aesthetic, dplyr groups by columns. Tidying the data first is 80% of the work.

The golden rule: before analyzing, ask yourself "what is an observation here?". If the answer requires pointing at several columns at once ("the sales are in Q1, Q2 and Q3"), your data is wide and you probably need pivot_longer().

The pipe: reading code like a recipe

The pipe operator chains operations: it takes the result on the left and passes it as the first argument of the function on the right. Its value is psychological and practical at once: it turns code that reads inside-out into code that reads top-to-bottom, in the order you think the steps.

R
# Without a pipe: nested, reads inside-out (backwards!)
arrange(summarise(group_by(mtcars, cyl), mpg_avg = mean(mpg)), mpg_avg)

# With the native pipe |> — reads like a recipe, step by step
mtcars |>
  group_by(cyl) |>
  summarise(mpg_avg = mean(mpg)) |>
  arrange(mpg_avg)
# A tibble: 3 × 2 cyl mpg_avg <dbl> <dbl> 1 8 15.1 2 6 19.7 3 4 26.7

Read it out loud: "take mtcars, then group by cylinders, then summarize the mean, then arrange". The |> mentally translates to "then". That's all the magic.

The native pipe |> vs magrittr's %>%

You'll see two pipes in the wild. The %>% from magrittr has existed since 2014 and popularized the style. The |> is native to R (since R 4.1) and today is the recommended one because it doesn't depend on any package. Differences worth knowing:

Aspect|> native%>% magrittr
Requires a packageNo, it's part of RYes (comes with tidyverse)
Placeholder_ (with named arg). (in any position)
SpeedSlightly fasterMinimal overhead

For 95% of cases they're interchangeable. The difference appears when the result doesn't go as the first argument. With magrittr you use the dot; with the native one, the _ placeholder requires the argument to be named:

R
# magrittr: the "." can go in any position
model <- mtcars %>% lm(mpg ~ wt, data = .)

# native: the "_" only with a named argument
model <- mtcars |> lm(mpg ~ wt, data = _)

Essential shortcut: in RStudio, Ctrl+Shift+M (or Cmd+Shift+M on Mac) inserts the pipe. You can choose which one it inserts under Tools › Global Options › Code › Use native pipe operator. Turn it on to type |>.

Which to use? For new code, |>: it doesn't depend on packages and is the standard going forward. You'll find plenty of %>% in tutorials and existing projects; understand it, but you don't need to migrate it if it already works.

Why the tidyverse made R accessible

Before the tidyverse, base R was powerful but cryptic: nested brackets, apply with obscure arguments, inconsistent names. The tidyverse offered a coherent vocabulary of verbs, a predictable (tidy) data structure and the pipe to chain them. The result: someone who never programmed can read data |> filter(salary > 2000) |> summarise(average = mean(age)) and understand it almost immediately. That readability is why R jumped from a statistical niche to an everyday tool for economists, data journalists and analysts.

dplyr: manipulating data

dplyr is not a collection of loose functions: it is a grammar. Every verb obeys the same contract: a data frame goes in and a data frame comes out. Because the output is the same type as the input, you can chain verbs with the pipe |> and read a complex transformation as a sentence: "take the sales, keep the ones from this quarter, group them by region, and summarise the total". If you internalize six verbs and that contract, you cover 90% of tabular data manipulation. Current version: dplyr 1.2.

The mental model: a data frame in, a data frame out

That uniformity is the key idea. In base R you jump between [], $, subset(), aggregate(), and apply(), each with its own syntax. In dplyr, every verb takes the table as its first argument and returns another table, so the result of one step can always feed the next. The pipe |> simply moves that object to the right: x |> f(y) is f(x, y).

VerbActionDoes it change rows or columns?
filter()Keep rows that meet a conditionReduces rows
select()Choose, rename, or reorder columnsChooses columns
mutate()Create or transform columnsAdds columns
arrange()Sort rowsReorders rows
summarise()Collapse many rows into a summaryReduces to 1 row (per group)
group_by()Mark groups to operate part by partDoesn't change data, changes context

Row verbs vs. column verbs: filter() and arrange() operate on rows (they use column values to decide); select() and mutate() operate on columns. summarise() is the only one that collapses: it turns N rows into 1 (or 1 per group).

filter(): choosing rows with conditions

Inside filter() you can refer to the columns by name. Combine conditions with & (and), | (or), and ! (not). Separating conditions with commas is equivalent to an &.

R · dplyr
library(dplyr)

sales <- tibble(
  region  = c("North","North","South","South","Central"),
  amount  = c(120, 340, NA, 80, 260),
  channel = c("web","store","web","web","store")
)

# Commas = "and". These two lines are equivalent:
sales |> filter(region == "North", amount > 100)
sales |> filter(region == "North" & amount > 100)

# explicit "or" with |
sales |> filter(channel == "web" | amount > 300)

The == trap with NA: any comparison with NA returns NA, not FALSE, and filter() discards rows where the condition is not explicitly TRUE. That's why filter(amount == NA) never works: the answer to "is this value equal to an unknown?" is "I don't know". To capture missing values use is.na(amount); to exclude them, !is.na(amount). Note too that in the example the South row with amount = NA disappears from amount > 100 even if you didn't expect it to.

select(): choosing columns with helpers

select() goes far beyond listing names. It accepts ranges with :, exclusion with !, and a family of selection helpers that choose by pattern. This makes your code survive when the data changes.

R · dplyr
customers |> select(id, name)                  # by name
customers |> select(id:city)                    # range of columns
customers |> select(!c(phone, email))           # all EXCEPT these
customers |> select(starts_with("date_"))       # prefix
customers |> select(ends_with("_2026"))         # suffix
customers |> select(contains("balance"))        # substring
customers |> select(where(is.numeric))          # by column type

where(is.numeric) is gold: it selects columns by a property, not by their name. If tomorrow you add another numeric column, it comes in on its own. Combine it: select(id, where(is.numeric)) keeps the identifier and everything numeric.

mutate(): creating and transforming columns

mutate() adds new columns (or overwrites existing ones by using the same name). You can refer to columns you just created within the same call, and control where they appear with .before / .after. For conditional logic use if_else() (two branches, strict about types) or case_when() (multiple branches).

R · dplyr
sales |>
  mutate(
    with_vat = amount * 1.19,
    segment  = if_else(amount >= 200, "high", "low"),
    grade    = case_when(
      amount >= 300 ~ "A",
      amount >= 150 ~ "B",
      is.na(amount) ~ "no data",
      .default     = "C"
    ),
    .after = amount           # place the new ones right after 'amount'
  )
region amount with_vat segment grade channel North 120 142.8 low C web North 340 404.6 high A store South NA NA low no data web South 80 95.2 low C web Central 260 309.4 high B store

if_else() vs ifelse(): dplyr's if_else() requires both branches to be the same type and handles NA predictably (that's why the "low" in the NA row: the condition NA >= 200 yields NA, and without a missing argument the result would be NA; here you see the nuance of propagating missing values). Base R's ifelse() is looser and can silently coerce types. In dplyr, prefer if_else().

arrange(): sorting rows

R · dplyr
sales |> arrange(amount)              # ascending (NAs go to the end)
sales |> arrange(desc(amount))        # descending
sales |> arrange(region, desc(amount)) # by region, then by amount ↓

group_by() + summarise(): the aggregation machine

This is where the analytical magic happens. group_by() doesn't change the data: it only marks how to split it. The verb that follows operates within each group. With summarise(), each group collapses to a single row. Typical functions: n() (row count of the group), sum(), mean(), n_distinct().

R · dplyr
sales |>
  group_by(region) |>
  summarise(
    n          = n(),
    total      = sum(amount, na.rm = TRUE),
    avg_ticket = mean(amount, na.rm = TRUE)
  )
# A tibble: 3 × 4 region n total avg_ticket <chr> <int> <dbl> <dbl> 1 Central 1 260 260 2 North 2 460 230 3 South 2 80 80

na.rm = TRUE is not optional: sum() and mean() return NA if the group contains a single missing value. In the South region, without na.rm the total would be NA. Always decide consciously whether you want to ignore missing values or whether their presence is itself a signal.

across(): the same operation over several columns

When you want the average of five columns, don't repeat five lines. across() applies one (or several) functions to a set of columns chosen with the same helpers as select(). It's the piece that makes dplyr truly scalable.

R · dplyr
# Average of all numeric columns, by region
sales |>
  group_by(region) |>
  summarise(across(where(is.numeric), \(x) mean(x, na.rm = TRUE)))

# Several functions at once, with automatic column names
sales |>
  group_by(region) |>
  summarise(across(where(is.numeric),
                   list(mean = \(x) mean(x, na.rm = TRUE),
                        max  = \(x) max(x,  na.rm = TRUE))))

Best practice: the modern anonymous function \(x) mean(x, na.rm = TRUE) (R 4.1+) is clearer than the old ~ mean(.x, na.rm = TRUE) notation. Use it inside across() so it's explicit what each column receives.

slice_max() / slice_min(): the top-N per group

R · dplyr
# The largest sale in each region
sales |>
  group_by(region) |>
  slice_max(amount, n = 1)

# The 3 smallest sales overall
sales |> slice_min(amount, n = 3)

The nuance of ungroup()

A detail that produces silent bugs: after summarise(), dplyr removes the last grouping variable but may leave the result grouped by the remaining ones. If you grouped by two columns and summarise, the result is still grouped by the first, and the next mutate() will operate by groups without you asking for it.

R · dplyr
sales |>
  group_by(region, channel) |>
  summarise(total = sum(amount, na.rm = TRUE)) |>
  # Still grouped by 'region'! This mutate computes % WITHIN each region:
  mutate(pct = total / sum(total)) |>
  ungroup()   # returns the data frame to a flat state

Defensive rule: when you finish an aggregation pipeline and are about to keep operating, close it with ungroup(). It's cheap and it avoids phantom "per group" calculations. In dplyr 1.1+ you can also use summarise(..., .groups = "drop") to ungroup in one go.

Joins: combining related tables

You rarely have all the information in a single table. Joins attach columns from a table B to a table A using one or more key columns. What changes between join types is which rows are kept.

diagram
# A = sales (customer_id)      B = customers (customer_id, city)
#
# left_join(A, B)   all rows of A + columns of B  (NA if no match)
# inner_join(A, B)  only rows with a match in BOTH
# full_join(A, B)   all rows of both (NA where missing)
# anti_join(A, B)   rows of A WITHOUT a match in B   (orphan sales?)
# semi_join(A, B)   rows of A WITH a match in B, but WITHOUT bringing B's columns
#
#   left   [ full A | B where it exists ]
#   inner  [   only the intersection A∩B  ]
#   anti   [ A that doesn't appear in B (audit) ]
R · dplyr
sales <- tibble(customer_id = c(1, 2, 2, 4), amount = c(120, 340, 80, 260))
customers <- tibble(customer_id = c(1, 2, 3), city = c("Bogotá","Cali","Medellín"))

# Enrich each sale with the customer's city
left_join(sales, customers, by = "customer_id")

# Modern syntax with join_by (dplyr 1.1+), handy for keys with different names
left_join(sales, customers, by = join_by(customer_id))

# Are there sales from customers not in the master table? (quality control)
anti_join(sales, customers, by = "customer_id")
# left_join: the sale with customer_id = 4 ends up with city = NA customer_id amount city 1 120 Bogotá 2 340 Cali 2 80 Cali 4 260 NA # anti_join: reveals the orphan sale customer_id amount 4 260

anti_join and semi_join are underrated: they are your auditing tools. anti_join(sales, customers) answers "which sales point to nonexistent customers?" without bringing in junk columns. semi_join filters A by membership in B without duplicating rows if B has repeated keys, something a left_join would do.

Integrated example: regional economic report

  left_join(customers, by = join_by(customer_id)) |>  # bring in city and segment
  filter(!is.na(amount), year == 2026) |>             # clean and narrow down

  summarise(
    n_tx    = n(),

tidyr: reshaping data

dplyr transforms data that already has the right shape. tidyr is in charge of getting that shape. Its job is to reshape: go from wide to long and back, split and merge columns, and handle missing values. Behind it is the principle of tidy data: each variable is a column, each observation is a row, each value a cell.

The shape of data: wide vs. long

The same data can live in two shapes. In wide form, one unit takes up one row and its measurements spread across many columns (comfortable for a human to read). In long form, each measurement takes up its own row, with one column saying what was measured and another how much (comfortable for the machine and for ggplot2).

schematic
# WIDE (for humans)             # LONG (for analysis)
# region  jan    feb     mar    region  month    sales
# North    120    140     160   North   jan        120
# South     90    110     130   North   feb        140
#                                 North   mar        160
#                                 South   jan         90  ...

pivot_longer(): from wide to long

It's probably the tidyr function you'll use most, because ggplot2 almost always wants long data: one column for the x axis, one for the value, one for the color. pivot_longer takes a set of columns (with select's helpers) and stacks them into two new columns: names and values.

R · tidyr
library(tidyr)

sales_wide <- tibble(
  region = c("North","South"),
  jan    = c(120, 90),
  feb    = c(140, 110),
  mar    = c(160, 130)
)

long <- sales_wide |>
  pivot_longer(
    cols      = jan:mar,         # which columns to stack (range with :)
    names_to  = "month",         # name of the new labels column
    values_to = "sales"          # name of the new values column
  )
long
# A tibble: 6 × 3 region month sales <chr> <chr> <dbl> 1 North jan 120 2 North feb 140 3 North mar 160 4 South jan 90 5 South feb 110 6 South mar 130

Why ggplot loves it: now you can write ggplot(long, aes(month, sales, color = region)) + geom_line(aes(group = region)). In wide format you'd have three month columns and no clean way to map "month" to an axis. Long = one variable per column = direct mapping to aesthetics.

pivot_wider(): from long to wide

The inverse operation. It takes a names column and a values column and spreads them into many columns. It's what you do at the end of the analysis, to build a presentable table.

R · tidyr
wide <- long |>
  pivot_wider(names_from = month, values_from = sales)
wide
# A tibble: 2 × 4 region jan feb mar <chr> <dbl> <dbl> <dbl> 1 North 120 140 160 2 South 90 110 130

The rule of thumb: long for analysis, wide for humans. Work internally in long form (it makes grouping, filtering, plotting, and modeling easier) and pivot to wide only at the last step, when you produce a table for a report. pivot_longer is your entry point, pivot_wider your exit.

separate_wider_delim() and unite(): splitting and gluing columns

Often a single column hides several variables ("2026-06-18", "Ana Pérez"). separate_wider_delim() (the modern form, which replaces separate()) splits them by a delimiter; unite() does the opposite.

R · tidyr
df <- tibble(date = c("2026-06-18","2026-07-01"),
             name = c("Ana","Luis"), last_name = c("Pérez","Durán"))

# Split 'date' into three columns
df |> separate_wider_delim(
  date, delim = "-", names = c("year","month","day")
)

# Join name + last_name
df |> unite("full_name", name, last_name, sep = " ")
# separate_wider_delim → year month day name last_name 2026 06 18 Ana Pérez 2026 07 01 Luis Durán # unite → date full_name 2026-06-18 Ana Pérez 2026-07-01 Luis Durán

Watch out for types: after separate_wider_delim(), year, month, and day are text, not numbers (notice the "06" and "01" with a leading zero). Convert with mutate(year = as.integer(year)) if you're going to do arithmetic. For dates, it's almost always better to parse directly with lubridate rather than splitting.

Missing values: drop_na(), replace_na(), and fill()

R · tidyr
df |> drop_na()                       # remove rows with ANY NA
df |> drop_na(sales)                   # only if 'sales' is NA
df |> replace_na(list(sales = 0))      # NA in 'sales' → 0
df |> fill(region, .direction = "down") # carry the last non-NA value downward

Best practice: fill() solves a classic case: Excel sheets where the category is only written in the first row of each block and the following ones are left blank. fill(category, .direction = "down") fills those gaps with the value above. But use it with judgment: it only makes sense when the blank means "same as the previous one", not "missing data".

stringr: manipulating text

Real-world data arrives dirty: names with inconsistent capitalization, phone numbers in varied formats, codes buried in descriptions. stringr offers a family of consistent functions (all starting with str_, all taking the text vector as the first argument) to clean them up. Their superpower is regular expressions.

Why the str_ family is so convenient

In base R you have nchar, toupper, grepl, sub, gsub, regmatches... with different argument orders. stringr unifies everything: the text goes first (ideal for |>), the pattern after, and the names are predictable (str_detect, str_replace, str_extract).

Length, case, whitespace, and padding

R · stringr
library(stringr)
x <- c("Ana Pérez", "luis durán", "  CARLA  ")

str_length(x)              # 9 10 9  (counts spaces)
str_to_upper(x)            # "ANA PÉREZ" ...
str_to_lower(x)            # "ana pérez" ...
str_to_title(x)            # "Ana Pérez" "Luis Durán" "  Carla  "
str_trim(x)                # strips leading/trailing spaces: "CARLA"
str_squish(x)              # also collapses multiple internal spaces
str_pad("7", width = 4, pad = "0")   # "0007"  (handy for codes)

Detect, filter, and count

R · stringr
emails <- c("ana@gmail.com","luis@company.co","spam#broken")

str_detect(emails, "@")           # TRUE TRUE FALSE  (does it contain?)
str_subset(emails, "@")           # returns only the ones containing @
str_count("banana", "a")          # 3  (how many times it appears)
str_starts(emails, "ana")         # TRUE FALSE FALSE

str_detect fits perfectly inside filter(): customers |> filter(str_detect(email, "@company\\.co$")) keeps the corporate emails. It's the bridge between stringr and dplyr.

Replace, remove, and extract

R · stringr
str_replace("2026-06-18", "-", "/")       # "2026/06-18"  (only the 1st)
str_replace_all("2026-06-18", "-", "/")   # "2026/06/18"  (all)
str_remove_all("US$ 1,200", "[^0-9]")     # "1200"  (strips non-numeric)
str_extract("Order #4523", "\\d+")        # "4523"  (first block of digits)
str_extract_all("A12-B34", "[A-Z]\\d+")   # list("A12","B34")
str_split("a,b,c", ",")                   # list(c("a","b","c"))

Practical mini-intro to regular expressions (regex)

A regex is a pattern for describing text. Instead of searching for the exact word "4523", you search for "one or more digits". These are the building blocks that cover 90% of cases:

PatternMeansExample it captures
\\da digit (0-9)the 7 in "A7"
\\wletter, digit, or underscoreeach character of "id_12"
\\sa whitespace characterthe gap in "a b"
.any characteranything at all
+one or more of the previous\\d+ → "4523"
*zero or more of the previous"" or "aaa"
?zero or one (optional)"color" and "colour"
^start of the string^tel anchors to the start
$end of the string\\.co$ at the end
[abc]any of those characters[0-9], [A-Z]
[^abc]any EXCEPT those[^0-9] = non-digit

Why the double backslash in R? The real regex for "digit" is \d. But in R the backslash is already an escape character inside strings, so to put a literal \ into the text you write \\. Result: in your R code you write "\\d", R converts it to \d, and that \d is what the regex engine sees. Mnemonic: in R, every regex backslash is doubled. (In R 4.0+, raw strings r"(\d+)" avoid the doubling, but the \\d form is still the most common.)

Real example: cleaning a contacts database

R · stringr + dplyr
contacts <- tibble(
  name  = c("  ana pérez ", "LUIS  DURÁN", "Carla Gómez"),
  phone = c("tel: (300) 555-1212", "mobile 311.444.9090", "3155551234"),
  code  = c("FACT-A12", "FACT-B34", "REC-C99")
)

contacts |>
  mutate(
    name   = str_to_title(str_squish(name)),          # "Ana Pérez", "Luis Durán"
    phone  = str_remove_all(phone, "[^0-9]"),          # digits only: "3005551212"
    prefix = str_extract(code, "^[A-Z]+"),             # "FACT", "REC"
    number = str_extract(code, "\\d+")                 # "12", "34", "99"
  )
name phone code prefix number Ana Pérez 3005551212 FACT-A12 FACT 12 Luis Durán 3114449090 FACT-B34 FACT 34 Carla Gómez 3155551234 REC-C99 REC 99

Cleaning idea: to normalize phone numbers it's almost never worth describing the exact format (parentheses, dots, dashes vary too much). It's more robust to delete everything that isn't a digit with str_remove_all(phone, "[^0-9]") and keep the clean sequence. Fewer rules, more resistant to messiness.

lubridate: dates and times

Dates are treacherous data: different formats, months of varying length, leap years, time zones. In base R, parsing a date requires remembering codes like %Y-%m-%d from strptime, and the arithmetic is awkward. lubridate replaces all of that with functions whose name is the order of the date.

Why dates in base R are tedious

To read "18/06/2026" in base you need as.Date("18/06/2026", format = "%d/%m/%Y") and not slip up on the codes. To add a month you have to do arithmetic with seconds or manipulate components by hand. lubridate tackles both problems: parsing by name and natural arithmetic.

Parsing: the function name is the order

The ymd, dmy, mdy family (and their time versions ymd_hms) tells you the order in which year, month, and day come. lubridate takes care of the separators.

R · lubridate
library(lubridate)

ymd("2026-06-18")          # year-month-day  → 2026-06-18
dmy("18/06/2026")          # day/month/year  → 2026-06-18
mdy("06-18-2026")          # month-day-year  → 2026-06-18
ymd_hms("2026-06-18 14:30:00")   # with time → 2026-06-18 14:30:00 UTC

# It accepts varied formats without you changing the function:
dmy(c("18/06/2026", "1 jan 2026", "18-06-2026"))

Order matters: "03/04" is ambiguous. Is it April 3rd or March 4th? It depends on the origin of your data. dmy("03/04/2026") reads it as April 3rd; mdy("03/04/2026") as March 4th. Choosing the wrong function doesn't raise an error: it gives a silently wrong date. Find out your source's format before parsing, and verify with a couple of cases where day > 12.

Date arithmetic: periods vs. durations

Here is the subtlety that confuses everyone. There are two ways to measure time:

  • Periods (months(), days(), years()): time in units of the human calendar. "One month" respects the real length of the month.
  • Durations (dmonths(), ddays(), dyears()): time in exact seconds. A ddays(30) is always 30 × 86400 seconds, regardless of the calendar.
R · lubridate
ymd("2026-01-31") + months(1)   # 2026-02-28  (period: "one calendar month")
ymd("2026-01-31") + ddays(30)   # 2026-03-02  (duration: 30 exact days)

ymd("2024-02-29") + years(1)    # 2025-02-28  (period adjusts to non-leap year)
ymd("2024-02-29") + dyears(1)   # 2025-02-28 06:00:00 (duration: 365.25 days)

When to use which: for human business logic ("the invoice is due in 1 month", "renew in 1 year") use periods: months(1). For exact physical intervals ("how many seconds passed between two events?") use durations: ddays(). Confusing them produces off-by-one-day errors that show up right at the end of the month or in February.

Rounding dates: floor / ceiling / round

To aggregate by periods (sales per month, per week) you need to "flatten" each date to the start of its period. That's the job of floor_date.

R · lubridate
d <- ymd("2026-06-18")
floor_date(d, "month")     # 2026-06-01  (first day of the month)
ceiling_date(d, "month")   # 2026-07-01  (first day of the next month)
round_date(d, "month")     # 2026-07-01  (to the nearest)
floor_date(d, "week")      # start of the week

Example: aggregating a sales series by month

R · lubridate + dplyr
transactions <- tibble(
  date   = dmy(c("05/01/2026","28/01/2026","03/02/2026","19/02/2026")),

Visualization with ggplot2

ggplot2 is not "a plotting library": it is a grammar. Instead of memorizing a different function for every type of chart, you learn a handful of pieces that combine to produce virtually any figure. That is why ggplot2 remains the gold standard in R. Current version: ggplot2 4.0.

The mental model: layers that add up

The underlying idea (taken from Wilkinson's book The Grammar of Graphics) is that every chart breaks down into independent components you can swap out:

ComponentWhat it controlsTypical function
dataThe source data frameggplot(data, ...)
aestheticsWhich variable goes to which visual channelaes(x, y, color, ...)
geometriesThe shape that represents the datageom_point(), geom_bar()
scalesHow the data is translated to the channel (colors, axes)scale_x_log10()
facetsSplits into subpanels by groupfacet_wrap()
coordsThe coordinate systemcoord_flip()
themeThe appearance (typography, grid, background)theme_minimal()

The key: these pieces add up with +. Each line adds a layer on top of the previous one. You start with data and a mapping, and keep stacking geometries, scales and tweaks until the chart says what you want.

R · ggplot2

  geom_point(size = 2, alpha = 0.8) +                   # layer: points
  geom_smooth(method = "lm", se = FALSE) +              # layer: trend
  labs(title = "Fuel economy vs. engine displacement",
       x = "Displacement (L)", y = "Miles/gallon (highway)",
       color = "Type") +
  theme_minimal(base_size = 13)
# A scatter is drawn: each point is a car model, colored by type. # One regression line per type shows the negative relationship: the larger # the displacement, the lower the highway fuel economy.

aes(): the heart of the grammar

aes() connects variables from your data with visual channels: position (x, y), color, fill, size, shape, alpha. To "map" a variable to a channel means: let this visual attribute vary according to the values of that column.

Mapping vs. setting — the classic mistake: if a color depends on a variable, it goes inside aes(). If it is a constant color, it goes outside. Writing aes(color = "blue") does not paint things blue: it creates a literal variable "blue" and ggplot assigns it its own color (salmon red, typically) plus an absurd legend. The right way is geom_point(color = "blue"), outside aes().

R · aes
# MAP: the color varies according to the drv variable (drive train)
ggplot(mpg, aes(displ, hwy, color = drv)) + geom_point()

# SET: all points the same blue, no legend
ggplot(mpg, aes(displ, hwy)) + geom_point(color = "steelblue")

# The mistake: "steelblue" treated as data -> wrong color + useless legend
ggplot(mpg, aes(displ, hwy, color = "steelblue")) + geom_point()

Descriptive statistics

Before modeling anything, you have to look at the data. R was born precisely for that: to describe, summarize and test hypotheses. This section goes from the simple average to regression, always interpreting as an analyst would.

Measures of central tendency and dispersion

Two questions summarize almost any numeric variable: where is the center? (mean, median) and how spread out is the data? (standard deviation, variance, range). The center without the dispersion lies: two countries with the same average income can have opposite realities.

R
x <- mtcars$mpg
mean(x); median(x)                    # center
sd(x); var(x); IQR(x)                 # dispersion (sd = square root of var)
quantile(x, c(0.25, 0.5, 0.75))       # quartiles
range(x); diff(range(x))              # min/max and spread
summary(x)                            # everything at a glance
> mean(x); median(x) [1] 20.09062 [1] 19.2 > sd(x); var(x); IQR(x) [1] 6.026948 [1] 36.3241 [1] 7.375 > summary(x) Min. 1st Qu. Median Mean 3rd Qu. Max. 10.40 15.43 19.20 20.09 22.80 33.90

Mean vs. median: here the mean (20.09) exceeds the median (19.2), a sign of a tail toward high values (very efficient cars). When mean and median differ a lot, the distribution is skewed and the median better describes "the typical case". That is why the median income of a country is reported, not the average: a handful of millionaires inflates the mean.

The IQR (interquartile range, Q3 − Q1) is a robust measure of dispersion: it is not affected by extreme values, unlike the standard deviation. That is why the boxplot uses it as the box. Practical outlier rule: a data point outside [Q1 − 1.5·IQR, Q3 + 1.5·IQR] is flagged as an outlier.

Frequency tables

For categorical variables, the question is how many times each value appears and in what proportion.

R

> table(mtcars$cyl) 4 6 8 11 7 14 > round(prop.table(table(mtcars$cyl))*100, 1) 4 6 8 34.4 21.9 43.8 # 43.8% of the cars have 8 cylinders

Correlation

Correlation measures how well one variable is predicted from another, on a scale from −1 to 1. Near 0 = no linear relationship; near ±1 = strong linear relationship.

R · correlation
cor(mtcars$mpg, mtcars$wt)                       # Pearson (linear relationship)
cor(mtcars$mpg, mtcars$wt, method = "spearman")  # Spearman (rank-based)
cor(mtcars[, c("mpg","hp","wt")])                # correlation matrix
> cor(mtcars$mpg, mtcars$wt) [1] -0.8676594 # strong NEGATIVE relationship: more weight, less fuel economy > cor(mtcars[, c("mpg","hp","wt")]) mpg hp wt mpg 1.0000000 -0.7761684 -0.8676594 hp -0.7761684 1.0000000 0.6587479 wt -0.8676594 0.6587479 1.0000000

Pearson vs. Spearman: Pearson measures a linear relationship and assumes approximately normal data; it is sensitive to outliers. Spearman works with the ranks (orderings), so it captures any monotonic relationship —even a curved one— and resists extreme values. If you suspect non-linearity or have outliers, report Spearman.

Correlation ≠ causation. The fact that weight and fuel consumption correlate at −0.87 does not prove that weight causes the consumption (even if physically it does). Ice cream and drownings correlate, and neither causes the other: summer links them. Before claiming cause, ask about hidden variables and the direction of the effect.

Inference: from the sample to the conclusion

Descriptive statistics summarize what you have; inference asks whether the pattern is real or could have arisen by chance from sampling.

R · inference
t.test(mpg ~ am, data = mtcars)   # does fuel economy differ by transmission?
cor.test(mtcars$mpg, mtcars$wt)   # is the correlation significant?
model <- lm(mpg ~ wt + hp, data = mtcars)  # multiple linear regression
summary(model)
Coefficients: Estimate Std. Error t value Pr(>|t|) (Intercept) 37.22727 1.59879 23.285 < 2e-16 *** wt -3.87783 0.63273 -6.129 1.12e-06 *** hp -0.03177 0.00903 -3.519 0.00145 ** --- Residual standard error: 2.593 on 29 degrees of freedom Multiple R-squared: 0.8268, Adjusted R-squared: 0.8148 F-statistic: 69.21 on 2 and 29 DF, p-value: 9.109e-12

How to read that summary (as an analyst): the coefficient of wt = −3.88 means that, holding hp constant, each extra 1000 lb reduces fuel economy by ~3.9 mpg. The *** and the p-value < 2e-16 indicate that this effect is almost certainly not chance. The R² = 0.83 says the model explains 83% of the variation in fuel consumption — high, but remember: a high R² does not guarantee a good model (it may be overfit or missing an assumption). Always look at the residuals.

Honest nuances about the p-value: a small p says "it is unlikely to see this if there were no effect", not "the effect is large or important". With a lot of data, trivial effects come out "significant". Always report the size of the effect (the coefficient, its confidence interval) alongside the p-value, not the p alone.

For complete one-line descriptive summaries try skimr::skim(data) (types, missing values, quartiles and mini-histograms) or summarytools::dfSummary(data) (report-ready table). They save you dozens of loose summary() calls.

Automated reports: R Markdown and Quarto

Imagine a document where the text, the code and its results live together, and where regenerating everything with new data is a single click. That is a reproducible report. In 2026, Quarto is the recommended successor to R Markdown for achieving it.

The why: reproducibility

The traditional flow —run an analysis, copy the chart, paste it into Word, format by hand— breaks every time the data changes. A Quarto report inverts the logic: the document contains the code that produces its own numbers and figures. The data changes, you re-render, and everything updates on its own. There is no copy-paste to go wrong.

Quarto or R Markdown?

R Markdown is still perfectly valid and there are millions of documents in production with it. Quarto is the evolution: same concept, more consistent syntax, native multi-language support (R, Python, Julia, Observable) and first-class formats for websites, books, blogs and presentations. You do not need the rmarkdown package: Quarto is a separate program that works the same from RStudio, VS Code or the terminal. For new projects, use Quarto.

Anatomy of a Quarto document (.qmd)

A .qmd has three ingredients: the YAML (configuration, at the top between ---), the text in Markdown, and the chunks of code between ```{r}.

Quarto · report.qmd
---
title: "Sales report"
author: "Luis Durán"
date: today
format: html
execute:
  echo: false      # hide the code in the final report
  warning: false   # hide warnings
---

## Introduction

Text in **Markdown**, with free formatting.

```{r}

#| fig-cap: "Fuel economy vs. weight"
```

The #| options (read "hash pipe") configure each chunk from within: #| echo: false hides the code but shows the result; #| warning: false silences warnings; #| label: names the chunk (useful for referencing figures); #| eval: false shows the code without running it. The inline code `r ...` embeds an R value in the middle of a sentence, so the numbers in the text never become outdated.

Ideal if the team does not master the syntax but wants to take advantage of reproducibility.

#    map_dfr goes through each file, reads and combines by rows

That is where R pays off its learning curve.

SQL databases from R

Until now your data fit in a CSV that you opened with read_csv(). But in a real company the data lives on a database server: millions of rows that won't fit in your laptop's RAM, that update every second and that ten people share. R doesn't need to download all of that: it connects, asks for what it needs in the database's language (SQL) and brings back only the answer.

The mental model: R as client, the database as server

Think of a restaurant. You (R) don't walk into the kitchen to stir the pots; you ask the waiter for a specific dish and he brings you just that dish. The database is the kitchen —it stores and processes everything—, and R is the diner placing precise orders. The "language" the orders are placed in is SQL, and the "waiter" that translates between R and any database is a package called DBI.

DBI (Database Interface) is the key piece: it defines a standard interface —the same functions (dbConnect, dbGetQuery, dbDisconnect...)— no matter whether underneath there's PostgreSQL, MySQL, SQL Server or a SQLite file. You switch engines without rewriting your code; you only change the first connection line.

DBI + a driver: there are always two pieces. DBI is the common interface; the driver (RSQLite, RPostgres, RMariaDB...) knows how to talk to a specific engine. You install both, but your code almost always uses DBI:: functions.

Connecting and querying with SQLite

We use SQLite because it's an entire database in a single file: it requires installing no server and it's perfect for learning. The workflow has four verbs you'll memorize forever: connect, list, query, disconnect.

R · DBI
library(DBI)

# 1. Open the connection (here we create an example SQLite file)
con <- dbConnect(RSQLite::SQLite(), "store.sqlite")

# We load a test table so we have something to query
dbWriteTable(con, "customers",
  data.frame(
    id    = 1:5,
    name = c("Ana", "Beto", "Caro", "Dani", "Emi"),
    country  = c("CO", "MX", "CO", "AR", "CO"),
    purchases = c(12, 3, 27, 8, 5)
  )
)

# 2. What tables are in the database?
dbListTables(con)

# 3. Ask for ONLY what you need (this runs on the database, not in R)
customers_co <- dbGetQuery(con,
  "SELECT name, purchases FROM customers WHERE country = 'CO' ORDER BY purchases DESC")
customers_co

# 4. Close the connection (always!)
dbDisconnect(con)
[1] "customers" name purchases 1 Caro 27 2 Ana 12 3 Emi 5

Notice the key point: the WHERE country = 'CO' clause filtered inside the database. Even if the table had 50 million rows, R only received the 3 you asked for. That's the difference between hauling everything back and blowing up your memory, or bringing back exactly what's needed.

Always close the connection: every database supports a limited number of open connections. A script that opens connections and never closes them can exhaust the server's quota and leave the whole team without access. Use dbDisconnect(con) when you finish, or wrap it with on.exit() in your functions.

dbplyr: write dplyr, run SQL

Here comes the magic that makes R so comfortable with databases. With dbplyr you write exactly the same dplyr you already know —filter, group_by, summarise—, but instead of running on your machine, it gets translated to SQL and executed on the server. Only when you call collect() does the final result travel back to R.

R · dbplyr
library(dplyr)
library(dbplyr)

con <- dbConnect(RSQLite::SQLite(), "store.sqlite")

# tbl() points to a server table WITHOUT bringing it into R
sales <- tbl(con, "customers")

query <- sales |>
  filter(country == "CO") |>
  summarise(total_purchases = sum(purchases, na.rm = TRUE))

query            # doesn't run yet: it's a "lazy" query
collect(query)   # now it does: runs on the database and brings the result to R
# Source: SQL [1 x 1] # Database: sqlite ... total_purchases <dbl> 1 44

The keyword is lazy: while you chain verbs, dbplyr is only building the query; nothing runs until collect(). That lets you assemble complex transformations that the engine optimizes and runs on its hardware, not on your laptop.

Spy on the SQL with show_query(): don't trust the translation? Add show_query() at the end of the pipe and you'll see the exact SQL dbplyr will send. It's the best way to learn SQL: write dplyr, look at what it generates.

R · show_query()
sales |>
  filter(country == "CO") |>
  summarise(total_purchases = sum(purchases, na.rm = TRUE)) |>
  show_query()
<SQL> SELECT SUM(`purchases`) AS `total_purchases` FROM `customers` WHERE (`country` = 'CO')

Security: credentials and guarding against SQL injection

A beginner mistake with real consequences: pasting the database username and password straight into the script. That script ends up in Git, in an email, in a screenshot... and exposes the whole database. The professional rule is: credentials are never written in the code, they live in environment variables.

R · environment variables
# In a .Renviron file (NOT committed to Git):
#   DB_HOST=server.company.com
#   DB_USER=analyst
#   DB_PASS=secret_password

library(RPostgres)
con <- dbConnect(
  Postgres(),
  host     = Sys.getenv("DB_HOST"),
  user     = Sys.getenv("DB_USER"),
  password = Sys.getenv("DB_PASS"),
  dbname   = "production"
)

The .Renviron file (in the project root or in your home directory) is read when R starts. Add it to .gitignore and the credentials will never leave your machine.

The second risk is SQL injection. If you build a query by pasting in text that comes from a user, someone can sneak in malicious commands. Compare:

R · dbBind
user_country <- "CO'; DROP TABLE customers; --"   # malicious input

# ✗ DANGER: pasting text directly allows injection
bad <- paste0("SELECT * FROM customers WHERE country = '", user_country, "'")

# ✓ SAFE: parameterized query. The '?' is a placeholder, dbBind fills it in
res <- dbGetQuery(con,
  "SELECT * FROM customers WHERE country = ?",
  params = list(user_country))

With parameters (? + dbBind or the params argument), the user's value is always treated as data, never as executable code. The database escapes it for you. It's the only correct way to put user input into a query.

Never use paste0() to build SQL with external data. That pattern is the number one cause of security breaches in data applications. Parameters always, no exceptions.

Dashboards

A dashboard is a single screen that answers "how is the business doing?" at a glance: a few big metrics up top, a couple of charts below. In R there are two routes depending on who will use it and how: static (Quarto Dashboard, an HTML you render and share) or interactive (Shiny, a live app where the user filters and explores, which you'll see in Level 3).

The mental model: report vs application

The question that decides everything is simple: does the user need to interact, or just look? If your boss opens a report on Monday with the week's numbers and touches nothing, that's a report: a static Quarto dashboard. If instead they need to pick a region from a menu, drag a date slider and watch everything recalculate live, that's an application: Shiny.

Rule of thumb: static = a "photo" of the data at render time; interactive = a "live query" that responds to every click. The static one is a file; the interactive one is a running server.

Quarto Dashboard (static, fast)

A Quarto dashboard is a normal .qmd file with format: dashboard. The structure is controlled with headings: each ## opens a row (or column), and inside you put valueboxes (those big boxes with a key metric) and charts. You render it once and get a self-contained HTML you email or upload to the web.

The valuebox is the first thing read. Reserve those boxes for the 2 or 4 metrics that truly matter (revenue, conversion, error). A dashboard with twelve valueboxes communicates nothing: when everything is important, nothing is.

Interactive charts: the best part of static

Even though the document is static (there's no server behind it), the charts inside can have client-side interactivity: tooltips on hover, zoom, tables with search. All of that travels packaged in the HTML. Two tools do 90% of the work:

  geom_point(size = 3)

The why matters: ggplotly() takes a ggplot you already know how to make and adds tooltips without rewriting anything —hover over a point and you see its exact values, something impossible in a fixed image—. And DT::datatable() turns a data frame into a table where the reader searches, sorts and paginates on their own. With those two, a "static" dashboard feels alive without needing Shiny.

Client-side vs server-side interactivity: plotly and DT run in the reader's browser (embedded JavaScript). They don't need R running behind them, which is why they work in an HTML you send by email. Shiny, on the other hand, needs a server with live R processing every click.

Functional programming

At its core, R is a functional language: functions are first-class objects. The purrr package takes this paradigm to its fullest expression.

map: iterating without loops

R · purrr
library(purrr)

map(1:3, \(x) x^2)             # returns a list
map_dbl(1:3, \(x) x^2)         # returns a double vector: 1 4 9
map_chr(letters[1:3], toupper)# "A" "B" "C"

# map2 and pmap for multiple arguments
map2_dbl(1:3, 4:6, \(a, b) a + b)   # 5 7 9

# Fit a model per group
mtcars |>
  split(mtcars$cyl) |>
  map(\(df) lm(mpg ~ wt, data = df)) |>
  map(coef)

The suffixes indicate the output type: map() → list, map_dbl() → double, map_chr() → character, map_lgl() → logical, map_df() → data frame. This makes your code predictable and safe.

reduce and pure functions

R
reduce(1:5, `+`)                  # 15  (accumulates)
reduce(list(df1, df2, df3), left_join, by = "id")   # joins N tables

# safely: captures errors without stopping the pipeline
safe_log <- safely(log)
map(list(1, -1, "a"), safe_log)

Pure functions: a function that, given the same arguments, always returns the same result and has no side effects. Pure functions are easy to test, parallelize, and reason about. Strive for them.

Code optimization

Before optimizing, measure. "Premature optimization is the root of all evil" (Knuth). Identify the real bottleneck and only then act.

Measuring (profiling)

R
library(bench)
mark(
  loop     = { s <- 0; for (i in 1:1e5) s <- s + i; s },
  vectoriz = sum(1:1e5)
)               # compares time and memory

system.time(sum(1:1e7))              # simple timing
profvis::profvis({ ... })            # visual line-by-line profiling

Strategies from highest to lowest impact

  1. Vectorize: replace loops with vectorized operations and map/apply functions.
  2. Pre-allocate memory; do not grow vectors inside loops.
  3. Use efficient structures: data.table for large data.
  4. Parallelize: furrr::future_map() uses multiple cores.
  5. Rewrite the critical parts in C++ with Rcpp (last resort).

80% of the gains come from 20% of the code. Do not optimize functions that run once; focus on what runs millions of times.

data.table

data.table is the go-to tool for large data in R: extremely fast and memory-efficient, with a concise syntax.

The dt[i, j, by] syntax

Read it as: "take the rows i, compute j, grouped by by".

R · data.table
library(data.table)
dt <- as.data.table(mtcars)

dt[mpg > 20]                              # filter rows
dt[, .(mean_mpg = mean(mpg)), by = cyl]   # summarize by group
dt[, weight_kg := wt * 453.6]             # create column BY REFERENCE (:=)
dt[order(-mpg)]                           # sort

# Chaining
dt[mpg > 15][, .(hp_max = max(hp)), by = gear]

The := operator modifies by reference (without copying the table), which saves an enormous amount of memory with datasets of millions of rows. That is the key to its speed.

dplyr or data.table? dplyr is more readable and sufficient for most cases; data.table wins when performance is critical (tens of millions of rows). The dtplyr package lets you write dplyr and run it on the data.table engine.

Machine Learning

R has a mature ML ecosystem. The modern framework is tidymodels, which unifies the workflow under a consistent grammar.

The tidymodels workflow

Split → preprocess (recipe) → specify the model → train (workflow) → evaluate → tune (tuning).

R · tidymodels
library(tidymodels)

# 1. Split the data (training / test)
set.seed(123)
split  <- initial_split(iris, prop = 0.8, strata = Species)
train  <- training(split); test <- testing(split)

# 2. Preprocessing recipe
recipe_spec <- recipe(Species ~ ., data = train) |>
  step_normalize(all_numeric_predictors())

# 3. Specify the model
model <- rand_forest(trees = 500) |>
  set_engine("ranger") |>
  set_mode("classification")

# 4. Workflow + training
wf <- workflow() |> add_recipe(recipe_spec) |> add_model(model)
fit <- wf |> fit(data = train)

# 5. Evaluate
pred <- predict(fit, test) |> bind_cols(test)
accuracy(pred, truth = Species, estimate = .pred_class)

Overfitting: never evaluate on the same data you trained on. Use cross-validation (vfold_cv()) and hold out a test set the model does not see until the very end.

Unsupervised models

They find structure in data without labels: clustering and dimensionality reduction.

R
# K-means: group into k clusters
data <- scale(iris[, 1:4])            # scaling is crucial!
km <- kmeans(data, centers = 3, nstart = 25)
km$cluster

# Hierarchical clustering
d <- dist(data)
hc <- hclust(d, method = "ward.D2")
plot(hc)

# PCA: reduce dimensions
pca <- prcomp(iris[, 1:4], scale. = TRUE)
summary(pca)        # variance explained per component

Scale your variables before clustering or PCA. If one variable is in the thousands and another in decimals, the first will dominate the distance. Use scale().

To choose the number of clusters, use the elbow method or the silhouette (factoextra::fviz_nbclust()). There is no single right answer; business context rules.

Time series

Data ordered in time (monthly sales, prices, sensors). The modern ecosystem is Rob Hyndman's fpp3/fable.

Key concepts: trend (long-term direction), seasonality (repeating patterns), and autocorrelation (each value depends on the previous ones). That is why you cannot use random cross-validation: temporal order matters.

Shiny: interactive web apps

Shiny turns your R analysis into an interactive web application, with no HTML or JavaScript required. It is the bridge between the analysis and the end user.

The two parts: UI and server

R · app.R
library(shiny)

ui <- fluidPage(
  titlePanel("mtcars explorer"),
  sidebarLayout(
    sidebarPanel(
      sliderInput("hp", "Minimum horsepower (hp):",
                  min = 50, max = 350, value = 100)
    ),
    mainPanel(plotOutput("plot"))
  )
)

server <- function(input, output) {
  output$plot <- renderPlot({
    library(ggplot2)
    data <- subset(mtcars, hp >= input$hp)    # reactive to input$hp
    ggplot(data, aes(wt, mpg)) + geom_point(size = 3) + theme_minimal()
  })
}

Reactivity: the heart of Shiny. When the user moves the slider, everything that depends on input$hp is recalculated automatically. Key concepts: reactive(), observeEvent(), eventReactive().

Best practices and project architecture

A professional project is readable, reproducible, and maintainable. Structure matters as much as the code.

Recommended project structure

project/
my_project/
├── my_project.Rproj     # RStudio project
├── README.md            # what it does and how to run it
├── renv.lock            # package versions
├── data/
│   ├── raw/             # raw data (never modified)
│   └── processed/       # generated clean data
├── R/                   # reusable functions
├── scripts/             # 01_cleaning.R, 02_analysis.R...
├── output/              # plots, tables, models
└── reports/             # Quarto reports

Key principles

  • Reproducibility: anyone should be able to clone and run it. Use renv and relative paths (here).
  • Consistent style: follow the tidyverse style guide; automate it with styler and lintr.
  • Don't repeat yourself (DRY): if you copy code three times, turn it into a function.
  • Separate raw data from processed data: raw data is sacred; never overwrite it.
  • Document and test what matters.

For complex, reproducible workflows, the targets package orchestrates your pipeline: it only recomputes what changed, like a Makefile for R.

The problem

Predict which customers will leave a service (churn) and deploy the model as an API + dashboard. It is a classic use case and highly valued in industry.

The 7 stages

  1. Ingestion: load data (CSV/SQL) within a reproducible project.
  2. Cleaning and EDA: tidyverse + visualization to understand the problem.
  3. Feature engineering: create predictive variables with recipes.
  4. Modeling: train and compare models with tidymodels + cross-validation.
  5. Evaluation: appropriate metrics (recall, AUC) and interpretability (vip, SHAP).
  6. Deployment: serve the model with plumber + a Shiny dashboard.
  7. Documentation: Quarto report and README, all version-controlled on GitHub.
R · pipeline skeleton
library(tidymodels)

# Data (example: telco churn)
data <- read_csv("data/raw/churn.csv") |>
  mutate(churn = factor(churn))

set.seed(42)
split <- initial_split(data, prop = 0.8, strata = churn)
folds <- vfold_cv(training(split), v = 5, strata = churn)

recipe <- recipe(churn ~ ., data = training(split)) |>
  step_rm(id_cliente) |>
  step_impute_median(all_numeric_predictors()) |>
  step_dummy(all_nominal_predictors()) |>
  step_normalize(all_numeric_predictors())

model <- boost_tree(trees = 500, learn_rate = 0.05) |>
  set_engine("xgboost") |> set_mode("classification")

wf  <- workflow() |> add_recipe(recipe) |> add_model(model)
res <- fit_resamples(wf, folds, metrics = metric_set(roc_auc, recall))
collect_metrics(res)

final <- last_fit(wf, split)
collect_metrics(final)         # performance on the test set
