What is R and what is it for?

R is a free, open-source programming language and environment designed specifically for statistical analysis, data science, and visualization.

ConceptEcosystemUse cases

Concept

R was born in 1993, created by Ross Ihaka and Robert Gentleman at the University of Auckland as a free implementation of the S language. Today it is maintained by the R Core Team and a huge worldwide community. The most recent stable release is R 4.6.0 ("Because it was There", April 2026).

Unlike a general-purpose language, R was designed from the ground up to work with data: vectors, tables, statistical models, and plots are first-class citizens of the language.

Use cases

  • Statistical analysis and econometrics (regressions, hypothesis tests, mixed models).
  • Data science and Machine Learning (classification, clustering, prediction).
  • High-quality visualization with ggplot2, ready for scientific publications.
  • Reproducible reports and dashboards with Quarto, R Markdown, and Shiny.
  • Bioinformatics, finance, public health, academic research, and BI.

R or Python?

Both are excellent. R shines in statistics, exploratory visualization, and communicating results; Python excels in software production, deep learning, and general-purpose tasks. It's not a war: many professionals use both. If your focus is data analysis and statistics, R is one of the best investments you can make.

Key idea: in R almost everything is a vector and almost everything is done with functions. Internalizing those two ideas makes the rest of the language fall into place naturally.

Your first line of R

R
# Everything after # is a comment
print("Hello, R!")
2 + 2
mean(c(10, 20, 30))   # average of three numbers
[1] "Hello, R!" [1] 4 [1] 20

Best practice: comment your code from day one. Your future self will thank you.

Practice exercise

Without installing anything yet, write down on paper what you think these would return: 3 * 4, sqrt(16), and c(1, 2, 3) * 2.

View solution

3 * 4 → 12. sqrt(16) → 4. c(1, 2, 3) * 2 → 2 4 6 (R multiplies each element: this is called vectorization).

Installing R and RStudio

You need two things: R (the language engine) and RStudio (the environment where you'll write comfortably). Both are free.

Step 1 — Install R (CRAN)

Download R from cran.r-project.org. CRAN is the official repository. Choose your operating system and download the most recent base version (R 4.6.x).

  • Windows: Download R for Windows → base → Download R 4.6.x for Windows.
  • macOS: download the .pkg that matches your chip (Apple Silicon or Intel).
  • Linux: use your package manager (r-base) or follow the CRAN instructions for your distribution.

Step 2 — Install RStudio (Posit)

Download RStudio Desktop (free) from posit.co. RStudio automatically detects your R installation.

Correct order: install R first and RStudio afterwards. RStudio is only the interface; it does not include the language.

Step 3 — Verify the installation

R · Console
R.version.string   # shows the installed version
sessionInfo()      # full session information
[1] "R version 4.6.0 (2026-04-24)"

Cloud alternative

If you can't install anything, use Posit Cloud: RStudio running in the browser, no installation required. Ideal for getting started right away or working from multiple machines.

Data types

Every value in R has a type. Knowing them prevents subtle bugs in calculations and comparisons.

Concept

TypeExampleDescription
numeric / double3.14Real numbers (the default type for numbers).
integer5LIntegers. The L forces the integer type.
character"text"Text strings, in quotes.
logicalTRUE / FALSEBoolean values.
complex2+3iComplex numbers (rarely used).

Example: inspecting types

R
class(3.14)      # "numeric"
class(5L)        # "integer"
class("hello")   # "character"
class(TRUE)      # "logical"

# Explicit conversion (coercion)
as.integer("42")     # 42
as.numeric("3.5")    # 3.5
as.character(100)    # "100"
as.logical(0)        # FALSE
[1] "numeric" [1] "integer" [1] "character" [1] "logical"

Special values

  • NA — missing value (Not Available).
  • NULL — absence of an object (zero length).
  • NaN — "not a number" (e.g. 0/0).
  • Inf / -Inf — infinity (e.g. 1/0).

Common mistake: comparing with NA using ==. x == NA always returns NA, not TRUE/FALSE. Always use is.na(x).

Silent coercion: if you mix types, R "promotes" everything to the most general one. c(1, "a", TRUE) becomes text. Check with class() if something doesn't add up.

Practice exercise

Predict the type of c(1L, 2.5), of c(TRUE, 3), and of c(1, "two"). Verify with class().

View solution

c(1L, 2.5) → numeric (the integer is promoted to double). c(TRUE, 3) → numeric (TRUE counts as 1). c(1, "two") → character. The coercion hierarchy is: logical → integer → double → character.

Variables and objects

A variable is a name that points to a value. In R, everything you create (a number, a table, a model) is an object.

Assignment

R
age <- 30           # the preferred operator in R
name = "Ana"        # valid, but less idiomatic
3.1416 -> pi_approx # right-hand assignment (rare)

age
name
[1] 30 [1] "Ana"

<- or =? The R convention is to use <- for assigning variables and reserve = for arguments inside functions. Shortcut: Alt+-.

Naming rules

  • Start with a letter (or a dot). Names may contain letters, numbers, . and _.
  • R is case-sensitive: Age ≠ age.
  • Avoid names of existing functions (mean, data, c).

Best practice: use snake_case (total_price), descriptive names, and a single language throughout. The tidyverse style guide is the de facto standard.

Inspecting the environment

R
ls()              # list the objects you have created
rm(age)           # remove an object
rm(list = ls())   # remove EVERYTHING in the environment (use with care)
str(name)         # structure/type of an object
Practice exercise

Create variables base = 10 and height = 4, compute the area of a triangle, and store it in area. Print the result.

View solution
R
base <- 10
height <- 4
area <- (base * height) / 2
area   # 20

Operators

Operators are symbols that perform arithmetic, comparison, and logical operations.

R
7 + 3     # 10  addition
7 - 3     # 4   subtraction
7 * 3     # 21  multiplication
7 / 3     # 2.333...  division
7 %% 3    # 1   modulo (remainder)
7 %/% 3   # 2   integer division
2 ^ 3     # 8   exponentiation
R
5 == 5    # TRUE   equal to
5 != 3    # TRUE   not equal to
5 > 3     # TRUE   greater than
5 <= 2    # FALSE  less than or equal to
"a" < "b" # TRUE   alphabetical order
R
TRUE & FALSE    # FALSE  AND (vectorized)
TRUE | FALSE    # TRUE   OR (vectorized)
!TRUE           # FALSE  negation
# && and || evaluate only the first element (for if)
(5 > 3) && (2 < 4)   # TRUE

One vs. two: & and | operate element by element on vectors; && and || return a single value and are used in if conditions. Since R 4.3, using && with vectors of length >1 throws an error.

Classic mistake: using = instead of == in a comparison. x = 5 assigns; x == 5 compares.

Practice exercise

Is 2025 a leap year? A year is a leap year if it is divisible by 4 and (not by 100, or also by 400). Write it as a single logical expression.

View solution
R
year <- 2025
(year %% 4 == 0) & (year %% 100 != 0 | year %% 400 == 0)
# FALSE  (2025 is not a leap year)

Vectors

The vector is R's most fundamental structure: an ordered sequence of values of the same type. Mastering vectors is mastering R.

Creating vectors

R
v <- c(10, 20, 30, 40)     # c() = combine
1:5                         # 1 2 3 4 5 (sequence)
seq(0, 1, by = 0.25)        # 0.00 0.25 0.50 0.75 1.00
rep("a", times = 3)         # "a" "a" "a"
length(v)                   # 4
[1] 10 20 30 40

Indexing (it starts at 1!)

R
v[1]            # 10  first element
v[c(1, 3)]      # 10 30
v[-1]           # 20 30 40  (everything except the first)
v[v > 20]       # 30 40  logical indexing
v[2] <- 99      # modify an element

Very common mistake: in R, indices start at 1, not 0 (unlike Python). v[0] returns an empty vector.

Vectorization: R's superpower

R
prices <- c(100, 200, 300)
prices * 1.19          # applies VAT to all: 119 238 357
sqrt(prices)           # square root of each one
sum(prices)            # 600
mean(prices)           # 200
prices + c(10, 20, 30) # element-wise addition

Recycling: if the vectors have different lengths, R "recycles" the shorter one. c(1,2,3,4) + c(10,20) → 11 22 13 24. Handy, but be careful: if the length is not a multiple, R warns you.

Practice exercise

Given grades <- c(4.5, 3.0, 2.8, 4.9, 3.7): (a) how many grades are there? (b) what is the average? (c) how many are passing (≥ 3.0)?

View solution
R
grades <- c(4.5, 3.0, 2.8, 4.9, 3.7)
length(grades)        # 5
mean(grades)          # 3.78
sum(grades >= 3.0)    # 4  (TRUE counts as 1)

Matrices

A matrix is a vector with two dimensions: rows and columns, all of the same type. It is the foundation of linear algebra in R.

Creating and indexing

R
m <- matrix(1:6, nrow = 2, ncol = 3)
m
m[1, ]      # first row
m[, 2]      # second column
m[2, 3]     # element at row 2, column 3
dim(m)      # 2 3
t(m)        # transpose
[,1] [,2] [,3] [1,] 1 3 5 [2,] 2 4 6

By default, R fills the matrix by columns. Use byrow = TRUE to fill by rows.

Operations

R
A <- matrix(c(1,2,3,4), 2, 2)
A * 2          # multiplies each element
A %*% A        # MATRIX multiplication (not the same as *!)
rowSums(A); colSums(A)
rowMeans(A); colMeans(A)
solve(A)       # inverse matrix

Careful: A * B multiplies element by element; A %*% B is the matrix product. Mixing them up is a frequent mistake in linear algebra.

Practice exercise

Create a 3×3 matrix with the numbers 1 through 9 filled by rows and compute the sum of each column.

View solution
R
m <- matrix(1:9, nrow = 3, byrow = TRUE)
colSums(m)   # 12 15 18

Factors

Factors represent categorical variables: qualitative data with a finite number of categories (levels), such as sex, city, or education level.

Creating factors

R
sizes <- factor(c("M", "S", "L", "M", "S"))
levels(sizes)         # "L" "M" "S"  (alphabetical order)
table(sizes)          # frequency of each level

# ORDERED factor (the categories have a hierarchy)
level <- factor(c("low","high","medium"),
                levels = c("low","medium","high"),
                ordered = TRUE)
level[1] < level[2]   # TRUE  (low < high)

Internally, a factor is stored as integers with a "label" for each level. That is why they are efficient, and why statistical models understand them as categories.

Dangerous trap: converting a factor of numbers to numeric. as.numeric(factor(c("10","20"))) returns 1 2 (the internal codes), not 10 and 20. The correct way: as.numeric(as.character(f)).

Practice exercise

Create an ordered satisfaction factor with levels "poor" < "fair" < "good" and get its frequency table for the vector c("good","poor","good","fair").

View solution
R
sat <- factor(c("good","poor","good","fair"),
              levels = c("poor","fair","good"), ordered = TRUE)
table(sat)

Lists

The list is R's most flexible structure: it can hold elements of different types and lengths, even other lists. Models and many R results are lists.

Creating and accessing

R
person <- list(
  name = "Luis",
  age  = 28,
  hobbies = c("R", "chess", "movies")
)
person$name          # "Luis"  (access by name)
person[["age"]]      # 28      (double bracket = the element)
person["age"]        # a sublist containing age
person$hobbies[2]    # "chess"

One bracket vs. two: mylist[1] returns a list containing the first element; mylist[[1]] returns the contents of the first element. It is the #1 source of confusion with lists.

Why they matter

When you fit a model, the result is a list. Knowing how to navigate it gives you access to everything:

R
model <- lm(dist ~ speed, data = cars)
names(model)             # model components
model$coefficients       # estimated coefficients
str(model, max.level = 1)

View solution
R

mean(course$grades)   # 4.166...

Data Frames

The data frame is the star structure of data analysis: a table where each column is a variable (a vector) and each row is an observation. It is the equivalent of a spreadsheet.

Creating and exploring

R
df <- data.frame(
  name = c("Ana", "Beto", "Caro"),
  age  = c(25, 31, 28),
  city = c("Cali", "Bogotá", "Medellín")
)
head(df)        # first rows
str(df)         # structure: types and dimensions
summary(df)     # statistical summary by column
nrow(df); ncol(df); dim(df)
names(df)       # column names

Selecting and filtering (base R)

R
df$age                  # one column as a vector
df[, "name"]            # same thing
df[1, ]                 # complete first row
df[df$age > 27, ]       # rows that meet a condition
df[df$age > 27, "name"]      # combine filter and column
df$older <- df$age >= 30     # create a new column

R ships with datasets ready for practice: mtcars, iris, airquality. Type data() to see the list. We will use several throughout this guide.

Later on we will meet the tibble (from the tidyverse), a modern version of the data frame with cleaner printing and more predictable behavior.

Importing and exporting data

90% of projects start by reading data from a file. R can read practically any format.

R
# Base R
data <- read.csv("data.csv")

# readr (tidyverse) — faster and more robust
library(readr)
data <- read_csv("data.csv")            # comma separator
data <- read_csv2("data.csv")           # ; separator and decimal comma (Europe/LatAm)
R
library(readxl)
data <- read_excel("file.xlsx", sheet = "Sheet1")
excel_sheets("file.xlsx")      # see sheet names
R
library(haven)   # SPSS, Stata, SAS
read_sav("data.sav")
read_dta("data.dta")

library(jsonlite)
fromJSON("data.json")

readRDS("object.rds")   # R's native format
R
write_csv(data, "output.csv")
library(writexl); write_xlsx(data, "output.xlsx")
saveRDS(model, "model.rds")     # saves ANY R object

The #1 import error: file paths. Use RStudio projects and the here package (here::here("data","data.csv")) instead of absolute paths like "C:/Users/.../data.csv", which break on another machine.

Decimals and separators: in many countries the decimal mark is a comma and the field separator is ;. If numbers are read in as text, you probably need read_csv2() or to adjust locale().

Basic functions

A function takes arguments and returns a result. R ships with hundreds ready to use; knowing the most common ones speeds up all your work.

Functions you will use every day

CategoryFunctions
Numeric summariessum, mean, median, sd, var, min, max, range, quantile
Counting / orderinglength, sort, order, rev, unique, table
Roundinground, ceiling, floor, abs
Textpaste, paste0, nchar, toupper, substr
Logicalany, all, which, ifelse, is.na
R
x <- c(4, 8, 15, 16, 23, 42)
mean(x); median(x); sd(x)
round(mean(x), 2)              # 18  rounded to 2 decimal places
which(x > 15)                  # positions: 4 5 6
ifelse(x > 15, "high", "low")  # vectorized
paste("Value:", x)             # joins text and numbers

Arguments and NA

R
v <- c(10, NA, 30)
mean(v)                # NA  ← a single NA "contaminates" the result
mean(v, na.rm = TRUE)  # 20  ← na.rm removes the NAs before computing

Common mistake: getting NA when computing averages or sums. It is almost always because of missing data: add na.rm = TRUE.

Arguments can be passed by name or by position. round(3.14159, 2) = round(x = 3.14159, digits = 2). Naming them makes the code more readable.

Structured programming

So far you have run isolated commands. Structured programming lets you control the flow: make decisions, repeat, and reuse logic.

The three pillars

  • Sequence: instructions run in order.
  • Selection: if/else, switch — run code depending on conditions.
  • Iteration: for, while — repeat tasks.

Reproducible scripts

A good script reads from top to bottom and always produces the same result. Recommended structure:

R · analysis.R
# 1. Libraries
library(tidyverse)

# 2. Parameters and paths
threshold <- 100

# 3. Load data
data <- read_csv("sales.csv")

# 4. Transformation
# 5. Analysis / model
# 6. Results / export

Best practice: write code that fails loudly and early. A clear error at the start is far better than a silently wrong result at the end.

Conditionals

They let your code make decisions depending on whether certain conditions are met.

if / else if / else

R
grade <- 3.8
if (grade >= 4.0) {
  print("Excellent")
} else if (grade >= 3.0) {
  print("Pass")
} else {
  print("Fail")
}
[1] "Pass"

Vectorized ifelse() and switch()

R
# ifelse: applies the condition to an ENTIRE vector
grades <- c(4.5, 2.8, 3.2)
ifelse(grades >= 3.0, "Pass", "Fail")
# "Pass" "Fail" "Pass"

# switch: choose among several options
type <- "premium"
switch(type,
  basic   = 0,
  premium = 0.15,
  vip     = 0.30)            # 0.15

Classic mistake: using if with a multi-value vector. if (grades > 3) throws an error since R 4.3. For vectors, use ifelse() or, in the tidyverse, dplyr::case_when().

case_when() for multiple conditions

R · dplyr
library(dplyr)
case_when(
  grades >= 4.0 ~ "Excellent",
  grades >= 3.0 ~ "Pass",
  TRUE          ~ "Fail"   # default case
)
Practice exercise

View solution
R

          bmi <= 25  ~ "normal",

Loops

Loops repeat instructions. In R, however, there are often faster and more elegant alternatives: vectorization and the apply family.

for and while

R
for (i in 1:5) {
  print(i^2)
}

# while: repeats as long as the condition holds
n <- 1
while (n <= 3) {
  cat("Iteration", n, "\n")
  n <- n + 1
}

Classic antipattern: growing a vector inside the loop (result <- c(result, x)). It is painfully slow. Pre-allocate the size: result <- numeric(n) and then result[i] <- x.

The apply family (more idiomatic)

R
m <- matrix(1:6, nrow = 2)
apply(m, 1, sum)        # row sums (margin 1)
apply(m, 2, sum)        # column sums (margin 2)

sapply(1:5, function(x) x^2)   # returns a vector: 1 4 9 16 25
lapply(1:3, function(x) x*10)  # returns a list
mapply(function(a,b) a+b, 1:3, 4:6)  # 5 7 9

In the tidyverse, purrr::map() replaces lapply with a more consistent syntax. You will see it in the Advanced Level (functional programming).

Error handling

Robust code anticipates what can go wrong. R distinguishes three conditions: errors (stop execution), warnings (alert you), and messages (inform you).

tryCatch

R
safe_read <- function(path) {
  tryCatch(
    expr    = read.csv(path),
    error   = function(e) {
      message("Could not read: ", conditionMessage(e))
      NULL
    },
    warning = function(w) { message("Warning: ", conditionMessage(w)); NULL },
    finally = message("Read attempt finished.")
  )
}

Validate arguments and warn

R
root <- function(x) {
  if (!is.numeric(x))  stop("x must be numeric")
  if (any(x < 0))      warning("Negative values present: they produce NaN")
  sqrt(x)
}

# stopifnot: quick validation
pos_mean <- function(x) {
  stopifnot(is.numeric(x), length(x) > 0)
  mean(x)
}

The rlang package offers abort(), warn(), and inform() with richer messages. purrr::safely() and possibly() wrap functions so they never halt a pipeline.

Best practice: fail with actionable messages. "x must be numeric, got character" is far better than "Error in sqrt".

The pipe: the heart of the workflow

The pipe operator chains operations: the result on the left is passed as the first argument on the right. It makes code readable left to right, like a recipe.

R
library(tidyverse)

# Without the pipe (nested, hard to read)
arrange(summarise(group_by(mtcars, cyl), mpg_med = mean(mpg)), mpg_med)

# With the native pipe |> (R 4.1+) — reads top to bottom
mtcars |>
  group_by(cyl) |>
  summarise(mpg_med = mean(mpg)) |>
  arrange(mpg_med)

Two pipes: |> is R's native pipe (recommended today); %>% is magrittr's, still widely used. RStudio shortcut: Ctrl/Cmd+Shift+M.

Key concept: tidy data = each variable in a column, each observation in a row, each value in a cell. Almost the entire tidyverse assumes this format.

Functional programming

At its core, R is a functional language: functions are first-class objects. The purrr package takes this paradigm to its fullest expression.

map: iterating without loops

R · purrr
library(purrr)

map(1:3, \(x) x^2)             # returns a list
map_dbl(1:3, \(x) x^2)         # returns a double vector: 1 4 9
map_chr(letters[1:3], toupper)# "A" "B" "C"

# map2 and pmap for multiple arguments
map2_dbl(1:3, 4:6, \(a, b) a + b)   # 5 7 9

# Fit a model per group
mtcars |>
  split(mtcars$cyl) |>
  map(\(df) lm(mpg ~ wt, data = df)) |>
  map(coef)

The suffixes indicate the output type: map() → list, map_dbl() → double, map_chr() → character, map_lgl() → logical, map_df() → data frame. This makes your code predictable and safe.

reduce and pure functions

R
reduce(1:5, `+`)                  # 15  (accumulates)
reduce(list(df1, df2, df3), left_join, by = "id")   # joins N tables

# safely: captures errors without stopping the pipeline
safe_log <- safely(log)
map(list(1, -1, "a"), safe_log)

Pure functions: a function that, given the same arguments, always returns the same result and has no side effects. Pure functions are easy to test, parallelize, and reason about. Strive for them.

Code optimization

Before optimizing, measure. "Premature optimization is the root of all evil" (Knuth). Identify the real bottleneck and only then act.

Measuring (profiling)

R
library(bench)
mark(
  loop     = { s <- 0; for (i in 1:1e5) s <- s + i; s },
  vectoriz = sum(1:1e5)
)               # compares time and memory

system.time(sum(1:1e7))              # simple timing
profvis::profvis({ ... })            # visual line-by-line profiling

Strategies from highest to lowest impact

  1. Vectorize: replace loops with vectorized operations and map/apply functions.
  2. Pre-allocate memory; do not grow vectors inside loops.
  3. Use efficient structures: data.table for large data.
  4. Parallelize: furrr::future_map() uses multiple cores.
  5. Rewrite the critical parts in C++ with Rcpp (last resort).

80% of the gains come from 20% of the code. Do not optimize functions that run once; focus on what runs millions of times.

data.table

data.table is the go-to tool for large data in R: extremely fast and memory-efficient, with a concise syntax.

The dt[i, j, by] syntax

Read it as: "take the rows i, compute j, grouped by by".

R · data.table
library(data.table)
dt <- as.data.table(mtcars)

dt[mpg > 20]                              # filter rows
dt[, .(mean_mpg = mean(mpg)), by = cyl]   # summarize by group
dt[, weight_kg := wt * 453.6]             # create column BY REFERENCE (:=)
dt[order(-mpg)]                           # sort

# Chaining
dt[mpg > 15][, .(hp_max = max(hp)), by = gear]

The := operator modifies by reference (without copying the table), which saves an enormous amount of memory with datasets of millions of rows. That is the key to its speed.

dplyr or data.table? dplyr is more readable and sufficient for most cases; data.table wins when performance is critical (tens of millions of rows). The dtplyr package lets you write dplyr and run it on the data.table engine.

R has several object systems. Knowing them helps you understand how packages work and lets you create your own types.

SystemWhen

Web Scraping

When the data is not available through an API, you can extract it from the HTML with rvest. Always do it ethically and legally.

R · rvest
library(rvest)

page <- read_html("https://ejemplo.com/productos")

titles <- page |> html_elements(".producto-titulo") |> html_text2()

Ethics and legality: respect the site's robots.txt file and terms of service. Throttle your requests (Sys.sleep()), identify yourself with a user-agent, and never extract personal data without a legal basis. When an official API exists, use it.

For sites that load content with JavaScript, rvest is not enough: you will need browser automation with chromote or RSelenium.

Machine Learning

R has a mature ML ecosystem. The modern framework is tidymodels, which unifies the workflow under a consistent grammar.

The tidymodels workflow

Split → preprocess (recipe) → specify the model → train (workflow) → evaluate → tune (tuning).

R · tidymodels
library(tidymodels)

# 1. Split the data (training / test)
set.seed(123)
split  <- initial_split(iris, prop = 0.8, strata = Species)
train  <- training(split); test <- testing(split)

# 2. Preprocessing recipe
recipe_spec <- recipe(Species ~ ., data = train) |>
  step_normalize(all_numeric_predictors())

# 3. Specify the model
model <- rand_forest(trees = 500) |>
  set_engine("ranger") |>
  set_mode("classification")

# 4. Workflow + training
wf <- workflow() |> add_recipe(recipe_spec) |> add_model(model)
fit <- wf |> fit(data = train)

# 5. Evaluate
pred <- predict(fit, test) |> bind_cols(test)
accuracy(pred, truth = Species, estimate = .pred_class)

Overfitting: never evaluate on the same data you trained on. Use cross-validation (vfold_cv()) and hold out a test set the model does not see until the very end.

Supervised models

They learn from labeled data in order to predict. If the response is numeric → regression; if it is categorical → classification.

R
# Linear and logistic regression (base R)
lm(mpg ~ wt + hp, data = mtcars)
glm(am ~ mpg, data = mtcars, family = binomial)   # logistic

# Trees, random forest, boosting
library(ranger);  ranger(Species ~ ., data = iris)
library(xgboost)  # high-performance gradient boosting

Evaluation metrics

ProblemMetrics
RegressionRMSE, MAE, R²
ClassificationAccuracy, precision, recall, F1, AUC-ROC

The accuracy trap: with imbalanced classes (99% negatives), a model that always says "negative" achieves 99% accuracy and is useless. Look at recall, F1, and the confusion matrix.

Unsupervised models

They find structure in data without labels: clustering and dimensionality reduction.

R
# K-means: group into k clusters
data <- scale(iris[, 1:4])            # scaling is crucial!
km <- kmeans(data, centers = 3, nstart = 25)
km$cluster

# Hierarchical clustering
d <- dist(data)
hc <- hclust(d, method = "ward.D2")
plot(hc)

# PCA: reduce dimensions
pca <- prcomp(iris[, 1:4], scale. = TRUE)
summary(pca)        # variance explained per component

Scale your variables before clustering or PCA. If one variable is in the thousands and another in decimals, the first will dominate the distance. Use scale().

To choose the number of clusters, use the elbow method or the silhouette (factoextra::fviz_nbclust()). There is no single right answer; business context rules.

Time series

Data ordered in time (monthly sales, prices, sensors). The modern ecosystem is Rob Hyndman's fpp3/fable.

R · fable
library(fpp3)   # tsibble + fable + feasts

# Convert to a tsibble and visualize
series <- as_tsibble(AirPassengers)
series |> autoplot(value)

# Decompose into trend + seasonality + remainder
series |> model(STL(value)) |> components() |> autoplot()

# Model and forecast 24 periods
fit <- series |> model(arima = ARIMA(value), ets = ETS(value))
fit |> forecast(h = 24) |> autoplot(series)

Key concepts: trend (long-term direction), seasonality (repeating patterns), and autocorrelation (each value depends on the previous ones). That is why you cannot use random cross-validation: temporal order matters.

Shiny: interactive web apps

Shiny turns your R analysis into an interactive web application, with no HTML or JavaScript required. It is the bridge between the analysis and the end user.

The two parts: UI and server

R · app.R
library(shiny)

ui <- fluidPage(
  titlePanel("mtcars explorer"),
  sidebarLayout(
    sidebarPanel(
      sliderInput("hp", "Minimum horsepower (hp):",
                  min = 50, max = 350, value = 100)
    ),
    mainPanel(plotOutput("plot"))
  )
)

server <- function(input, output) {
  output$plot <- renderPlot({
    library(ggplot2)
    data <- subset(mtcars, hp >= input$hp)    # reactive to input$hp
    ggplot(data, aes(wt, mpg)) + geom_point(size = 3) + theme_minimal()
  })
}

shinyApp(ui, server)

Reactivity: the heart of Shiny. When the user moves the slider, everything that depends on input$hp is recalculated automatically. Key concepts: reactive(), observeEvent(), eventReactive().

For large apps, use Shiny modules (they encapsulate reusable UI+server pairs) and the {golem} framework to structure them as a package. The (free) book Mastering Shiny is the reference.

Best practices and project architecture

A professional project is readable, reproducible, and maintainable. Structure matters as much as the code.

Recommended project structure

project/
my_project/
├── my_project.Rproj     # RStudio project
├── README.md            # what it does and how to run it
├── renv.lock            # package versions
├── data/
│   ├── raw/             # raw data (never modified)
│   └── processed/       # generated clean data
├── R/                   # reusable functions
├── scripts/             # 01_cleaning.R, 02_analysis.R...
├── output/              # plots, tables, models
└── reports/             # Quarto reports

Key principles

  • Reproducibility: anyone should be able to clone and run it. Use renv and relative paths (here).
  • Consistent style: follow the tidyverse style guide; automate it with styler and lintr.
  • Don't repeat yourself (DRY): if you copy code three times, turn it into a function.
  • Separate raw data from processed data: raw data is sacred; never overwrite it.
  • Document and test what matters.

For complex, reproducible workflows, the targets package orchestrates your pipeline: it only recomputes what changed, like a Makefile for R.

★ Final project: End-to-end Data Science

Bring everything together: from raw data to a deployed model. This is the project you will showcase in your portfolio.

1 Concept2 Example3 Practice4 Challenge5 Project

The problem

Predict which customers will leave a service (churn) and deploy the model as an API + dashboard. It is a classic use case and highly valued in industry.

The 7 stages

  1. Ingestion: load data (CSV/SQL) within a reproducible project.
  2. Cleaning and EDA: tidyverse + visualization to understand the problem.
  3. Feature engineering: create predictive variables with recipes.
  4. Modeling: train and compare models with tidymodels + cross-validation.
  5. Evaluation: appropriate metrics (recall, AUC) and interpretability (vip, SHAP).
  6. Deployment: serve the model with plumber + a Shiny dashboard.
  7. Documentation: Quarto report and README, all version-controlled on GitHub.
R · pipeline skeleton
library(tidymodels)

# Data (example: telco churn)
data <- read_csv("data/raw/churn.csv") |>
  mutate(churn = factor(churn))

set.seed(42)
split <- initial_split(data, prop = 0.8, strata = churn)
folds <- vfold_cv(training(split), v = 5, strata = churn)

recipe <- recipe(churn ~ ., data = training(split)) |>
  step_rm(id_cliente) |>
  step_impute_median(all_numeric_predictors()) |>
  step_dummy(all_nominal_predictors()) |>
  step_normalize(all_numeric_predictors())

model <- boost_tree(trees = 500, learn_rate = 0.05) |>
  set_engine("xgboost") |> set_mode("classification")

wf  <- workflow() |> add_recipe(recipe) |> add_model(model)
res <- fit_resamples(wf, folds, metrics = metric_set(roc_auc, recall))
collect_metrics(res)

final <- last_fit(wf, split)
collect_metrics(final)         # performance on the test set
Final challenge

Complete the project with a real dataset (Telco Churn from Kaggle or similar): fit at least two models, compare them, pick the best one by AUC, interpret it with variable importance, expose a prediction endpoint via plumber, and build a Shiny dashboard where a user can enter a customer's data and get the churn probability. Document everything in Quarto and publish it on GitHub.
